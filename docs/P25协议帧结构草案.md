# P25 协议帧结构说明（草案）

> 版本：v0.1  
> 适用范围：当前工程中的 P25 Phase 1 离线解码链路，以及后续补全控制信道、语音和会话聚合时的结构参考。  
> 主要代码参考：`p25/constants.py`、`p25/dsp.py`、`p25/nid.py`、`p25/framing.py`、`p25/decoder.py`、`p25/hdu_decode.py`、`p25/lc_decode.py`、`p25/ldu2_decode.py`、`p25/session.py`。  
> 标准参考方向：APCO Project 25 Phase 1 Common Air Interface。本文先按当前工程已实现和已验证的结构展开，未完整实现的业务类型会标注为“待补全”。

## 1. 基本参数

### 1.1 空口与物理层参数

| 项目 | 数值 / 说明 |
|------|-------------|
| 协议名称 | APCO P25 Phase 1 |
| 频点 / 频段 | 协议本身不绑定固定 RF 频点；工程中的 `.rawiq` 是复基带文件，只保留相对中心频偏 |
| 典型信道带宽 | 12.5 kHz |
| 多址方式 | FDMA，Phase 1 不使用 TDMA |
| 调制方式 | C4FM 兼容的 4-level FSK 判决模型 |
| 符号率 | 4800 symbols/s |
| 每符号承载 | 2 bit / symbol |
| 原始比特率 | 9600 bit/s |
| 当前工程解码采样率 | 48 kHz |
| 当前工程每符号采样点 | 10 samples/symbol |
| 基本语音/头帧长度 | 864 symbols = 1728 raw bits |
| 基本语音/头帧时长 | 180 ms |
| Frame Sync | 48 bits = 24 symbols，固定值 `5575F5FF77FF` |
| NID | 64 logical bits，包含 NAC 和 DUID，并带 BCH 保护 |
| NAC | 12 bits，Network Access Code |
| DUID | 4 bits，Data Unit ID，决定 HDU/LDU/TDU/TSBK/PDU 等帧类型 |

### 1.2 当前工程使用的符号映射

当前工程把 P25 C4FM 符号按四电平 FSK 方式判决：

| Dibit | 判决电平 |
|-------|----------|
| `00` | `+1` |
| `01` | `+3` |
| `10` | `-1` |
| `11` | `-3` |

接收端处理：

```text
IQ
  -> scanner 统一 FSK frontend
  -> P25 Frame Sync NCC 检测
  -> 根据 FS 做极性、相位和幅度校准
  -> slice_symbols_to_bits()
  -> P25 frame bits
```

### 1.3 Frame Sync 与 DUID

P25 Frame Sync：

```text
FRAME_SYNC_HEX  = 5575F5FF77FF
FRAME_SYNC_BITS = 48 bits
FRAME_SYNC_SYMS = 24 symbols
```

当前工程识别的 DUID：

| DUID | 名称 | 类型 | 当前工程处理 |
|------|------|------|--------------|
| `0x0` | HDU | Header | 解码 Header Code Word |
| `0x3` | TDU | Terminator | 识别为结束帧 |
| `0x5` | LDU1 | Voice | 解码 Link Control |
| `0x7` | TSBK | Trunking Control | 识别 NID，控制字段待补全 |
| `0xA` | LDU2 | Voice | 解码 Encryption Sync |
| `0xC` | PDU | Packet Data | 识别 NID，数据字段待补全 |
| `0xF` | TDULC | Terminator with LC | 识别为结束帧，LC 待补全 |

## 2. 帧结构

一次 P25 Phase 1 语音传输可以按如下序列理解：

```text
Call / Transmission
    │
    ├─ HDU
    │     Header Data Unit，携带加密/算法/密钥 ID/TGID 等头部信息
    │
    ├─ LDU1 / LDU2 × N
    │     语音 Logical Link Data Unit 交替出现
    │     LDU1 携带 Link Control
    │     LDU2 携带 Encryption Sync
    │
    └─ TDU / TDULC
          结束传输；TDULC 可携带 LC，当前工程尚未完整解析
```

当前工程输出 PDU：

| PDU | 来源 | 当前含义 |
|-----|------|----------|
| `P25_NID` | 任意识别到的 P25 frame | 至少成功解析 FS + NID |
| `P25_HDU` | HDU | Header Code Word RS/Golay 解码成功 |
| `P25_LDU1` | LDU1 | Link Control 解码成功 |
| `P25_LDU2` | LDU2 | Encryption Sync 解码成功 |
| `P25_CALL` | TDU/TDULC 触发 | 会话聚合输出 |

### 2.1 通用帧头：FS + NID

当前工程所有 P25 帧都先恢复 FS + NID：

```text
P25 frame start

┌────────────────────────┬────────────────────────────────────┐
│ Frame Sync             │ NID air field                       │
│ 24 symbols / 48 bits   │ 33 symbols / 66 air bits             │
└────────────────────────┴────────────────────────────────────┘

NID air field 中含 1 个 status dibit：
66 air bits - 2 status bits = 64 logical NID bits
```

NID 逻辑字段：

```text
64-bit NID
┌────────────┬────────────┬──────────────────────────────┐
│ NAC        │ DUID       │ BCH parity / protection       │
│ 12 bits    │ 4 bits     │ 48 bits                       │
└────────────┴────────────┴──────────────────────────────┘
```

当前工程处理：

```text
FS + NID symbols
  -> slice_symbols_to_bits()
  -> extract_nid_bits()
       跳过 in-header status dibit
  -> decode_nid()
       BCH(63,16) 解码
       NAC / DUID 提取
  -> frame_info_from_nid()
       分类为 HDU/LDU1/LDU2/TDU/...
```

当前 NID PDU 关键字段：

| 字段 | 含义 |
|------|------|
| `extra.nac` | 12-bit NAC |
| `extra.duid` | 4-bit DUID |
| `extra.duid_name` | HDU/LDU1/LDU2/TDU 等名称 |
| `extra.valid_bch` | NID BCH 是否校验通过 |
| `extra.corrected` | 是否发生 BCH 纠错 |
| `extra.fs_start` | Frame Sync 起始样点 |
| `extra.sync_ncc` | Frame Sync 互相关峰值 |

### 2.2 HDU：Header Data Unit

HDU 总长度按当前工程恢复为：

```text
HDU = 864 symbols = 1728 raw bits = 180 ms
```

HDU 中当前工程解析 Header Code Word：

```text
HDU frame bits
  -> deinterleave_hdu()
  -> 36 组 Golay codeword
  -> Golay(24,6) decode
  -> 36 个 6-bit symbol
  -> RS(36,20,17)
  -> Header Code Word
```

Header Code Word 当前字段：

```text
Header Code Word = 120 bits

┌──────────────┬──────────┬──────────┬──────────┬──────────┐
│ MI           │ MFID     │ ALGID    │ KID      │ TGID     │
│ 72 bits      │ 8 bits   │ 8 bits   │ 16 bits  │ 16 bits  │
└──────────────┴──────────┴──────────┴──────────┴──────────┘
```

字段说明：

| 字段 | 当前工程字段名 | 含义 |
|------|----------------|------|
| MI | `mi` | Message Indicator |
| MFID | `hdu_mfid` | Manufacturer ID |
| ALGID | `algid` | Algorithm ID |
| KID | `kid` | Key ID |
| TGID | `hdu_tgid` / `dst` | Talkgroup ID |
| corrected | `hdu_golay_corrected` | Golay 纠错计数 |

当前 PDU：

```text
type      = P25_HDU
protocol  = P25
dst       = TGID
flco      = HDU
extra     = nac, duid, mi, mfid, algid, kid, tgid...
```

### 2.3 LDU1：语音 + Link Control

LDU1 总长度：

```text
LDU1 = 864 symbols = 1728 raw bits = 180 ms
```

LDU1 的主体是语音信息，Link Control 按固定位置穿插在 IMBE voice groups 之间。当前工程用位置表 `LC_HEXBIT_POSITIONS` 提取 LC 的 24 个 Hamming group：

```text
LDU1 frame bits
  -> deinterleave_lc()
  -> 24 × 10-bit Hamming group
  -> Hamming(10,6,3)
  -> 24 × 6-bit hexbit
  -> RS(24,12,13)
  -> 12 × 6-bit data hexbit
  -> 72-bit Link Control
  -> parse_link_control()
```

当前工程支持的 Link Control：

#### 2.3.1 Group Voice Channel User

```text
LCO = 0x00

┌──────────┬──────────┬──────────┬──────────────┬──────────────┐
│ LCO      │ MFID     │ SVC      │ TGID         │ SRC          │
│ 8 bits   │ 8 bits   │ 8 bits   │ 16 bits      │ 24 bits      │
└──────────┴──────────┴──────────┴──────────────┴──────────────┘
```

当前字段：

| 字段 | 当前工程字段名 | 含义 |
|------|----------------|------|
| LCO | `lco` | Link Control Opcode |
| MFID | `mfid` | Manufacturer ID |
| SVC | `svc` | Service Options |
| TGID | `tgid` / `dst` | Talkgroup ID |
| SRC | `src` | Source Unit ID |
| Emergency | `lc_emergency` | 从 SVC/保留位区域提取的紧急标记 |
| Call Type | `call_type` | `group` |

#### 2.3.2 Unit-to-Unit Voice

```text
LCO = 0x03

┌──────────┬──────────┬──────────┬──────────────┬──────────────┐
│ LCO      │ MFID     │ Octet2   │ Target ID    │ SRC          │
│ 8 bits   │ 8 bits   │ 8 bits   │ 24 bits      │ 24 bits      │
└──────────┴──────────┴──────────┴──────────────┴──────────────┘
```

当前字段：

| 字段 | 当前工程字段名 | 含义 |
|------|----------------|------|
| LCO | `lco` | `0x03` |
| MFID | `mfid` | Manufacturer ID |
| Target ID | `dst` | 目标 Unit ID |
| SRC | `src` | Source Unit ID |
| Call Type | `call_type` | `unit_to_unit` |

当前 PDU：

```text
type      = P25_LDU1
protocol  = P25
src/dst   = Link Control 地址
flco      = LDU1
extra     = nac, lco, mfid, svc, tgid, call_type, rs_ok...
```

### 2.4 LDU2：语音 + Encryption Sync

LDU2 总长度：

```text
LDU2 = 864 symbols = 1728 raw bits = 180 ms
```

当前工程解析 LDU2 的 Encryption Sync：

```text
LDU2 frame bits
  -> deinterleave_es()
  -> 24 × 10-bit Hamming group
  -> Hamming(10,6,3)
  -> 24 × 6-bit hexbit
  -> RS(24,16,9)
  -> 16 × 6-bit data hexbit
  -> 96-bit Encryption Sync
```

Encryption Sync 当前字段：

```text
Encryption Sync = 96 bits

┌──────────────┬──────────┬──────────┐
│ MI           │ ALGID    │ KID      │
│ 72 bits      │ 8 bits   │ 16 bits  │
└──────────────┴──────────┴──────────┘
```

当前 PDU：

```text
type      = P25_LDU2
protocol  = P25
flco      = LDU2
extra     = nac, es_mi, es_algid, es_kid, es_hamming_corrected...
```

### 2.5 TDU / TDULC

当前工程把 DUID 中的 `TDU` / `TDULC` 标记为 terminator：

```text
DUID 0x3 -> TDU
DUID 0xF -> TDULC
```

在 `P25SessionAssembler` 中：

```text
if frame_info.is_terminator:
  -> 如果当前有 active call
  -> 输出 P25_CALL
  -> reset session
```

`P25_CALL` 当前字段：

| 字段 | 含义 |
|------|------|
| `src` | 最近一次 LDU1 LC 中的源 ID |
| `dst` | 最近一次 LDU1 LC 中的目标 ID / TGID |
| `extra.nac` | NAC |
| `extra.duration_s` | 从第一帧到 terminator 的估计时长 |
| `extra.ldu_count` | 会话中观察到的 LDU 数量 |

TDULC 自身携带的 Link Control 当前尚未单独解码。

### 2.6 TSBK / PDU（待补全）

当前工程能通过 NID 识别：

```text
DUID 0x7 -> TSBK
DUID 0xC -> PDU
```

但 TSBK/PDU 的 payload 字段尚未下沉解析。后续需要补齐：

| 待补项 | 说明 |
|--------|------|
| TSBK payload FEC | 需要补齐控制信道块解码链路 |
| Trunking opcodes | 需要解析系统、站点、频道授权、通话授予等控制字段 |
| PDU packet data | 需要解析 packet data 头和分片 |
| 统一控制信道 PDU | 与 DMR CSBK / dPMR CCH 的统一输出形态对齐 |

## 3. 数据组织过程：从语音 bit 到空口信号

这一部分按“发送端生成空口”的方向描述。接收端解码基本是反向过程。

### 3.1 总体流程

```text
业务输入
  ├─ 语音：PCM / 麦克风音频
  └─ 控制：NAC、DUID、TGID、SRC、MFID、MI、ALGID、KID

语音路径
  PCM
    -> IMBE voice encoder
    -> voice codewords
    -> LDU1/LDU2 voice groups

控制路径
  HDU HCW / LDU1 LC / LDU2 ES
    -> FEC
    -> interleave
    -> 插入到 864-symbol frame 的固定位置

组帧
  Frame Sync
  NID
  HDU / LDU1 / LDU2 / TDU

调制
  dibits
    -> C4FM / four-level symbol
    -> 12.5 kHz RF channel
```

### 3.2 NID 生成过程

```text
NAC      12 bits
DUID      4 bits
----------------
info     16 bits

info
  -> BCH(63,16)
  -> 64-bit logical NID in current decoder representation
  -> air field 中插入 status dibit
  -> 33 symbols / 66 air bits
```

接收端：

```text
FS + NID air bits
  -> skip status dibit
  -> 64-bit logical NID
  -> BCH decode
  -> NAC / DUID
```

### 3.3 HDU 生成过程

```text
Header Code Word fields
  MI 72 bits
  MFID 8 bits
  ALGID 8 bits
  KID 16 bits
  TGID 16 bits
  -> 120 bits

120 bits
  -> 20 × 6-bit data symbols
  -> RS(36,20,17)
  -> 36 × 6-bit symbols
  -> Golay(24,6) protection per symbol group
  -> interleave into HDU frame
```

接收端对应：

| 阶段 | 函数 |
|------|------|
| HDU 位置提取 | `p25.dsp.deinterleave_hdu()` |
| Golay | `p25.fec.golay_24_6_decode()` |
| RS | `p25.fec.rs_36_20_17_decode()` |
| 字段解析 | `p25.hdu_decode.decode_hdu_hcw()` |

### 3.4 LDU1 Link Control 生成过程

```text
72-bit Link Control
  -> 12 × 6-bit data hexbit
  -> RS(24,12,13)
  -> 24 × 6-bit hexbit
  -> Hamming(10,6,3)
  -> interleave into LDU1 frame
```

接收端对应：

| 阶段 | 函数 |
|------|------|
| LC 位置提取 | `p25.dsp.deinterleave_lc()` |
| Hamming | `p25.fec.hamming_10_6_3_decode()` |
| RS | `p25.fec.rs_24_12_13_decode()` |
| LC 解析 | `p25.link_control.parse_link_control()` |

### 3.5 LDU2 Encryption Sync 生成过程

```text
96-bit Encryption Sync
  MI 72 bits
  ALGID 8 bits
  KID 16 bits

96 bits
  -> 16 × 6-bit data hexbit
  -> RS(24,16,9)
  -> 24 × 6-bit hexbit
  -> Hamming(10,6,3)
  -> interleave into LDU2 frame
```

接收端对应：

| 阶段 | 函数 |
|------|------|
| ES 位置提取 | `p25.dsp.deinterleave_es()` |
| Hamming | `p25.fec.hamming_10_6_3_decode()` |
| RS | `p25.fec.rs_24_16_9_decode()` |
| 字段解析 | `p25.ldu2_decode.decode_ldu2_es()` |

### 3.6 接收端离线解码数据流

```text
rawiq file
  -> common.io.read_rawiq()
  -> scanner 重采样到 48 kHz
  -> dmr.dsp.frontend()
       P25 当前复用统一 4FSK 前端
  -> p25.decoder.decode()
       find_frame_sync()
       recover_symbols_from_fs(FS+NID)
       extract_nid_bits()
       decode_nid()
       frame_info_from_nid()
       recover_full_frame() for HDU/LDU1/LDU2
       decode_hdu_hcw() / decode_ldu1_lc() / decode_ldu2_es()
       P25SessionAssembler.feed()
  -> radio.registry.deduplicate_pdus()
  -> scanner 输出文本或 JSON
```

主要中间产物：

| 中间产物 | 类型 | 含义 |
|----------|------|------|
| `iq` | `np.ndarray[complex]` | 原始复基带 IQ |
| `y` | `np.ndarray[float]` | 统一 FSK 鉴频输出 |
| `P25SyncCandidate` | dataclass | FS 起点、极性、NCC |
| `P25NID` | dataclass | NAC、DUID、BCH 状态 |
| `P25FrameInfo` | dataclass | DUID 对应的帧分类 |
| `HeaderCodeWord` | dataclass | HDU 头字段 |
| `LinkControl` | dataclass | LDU1 链路控制字段 |
| `EncryptionSync` | dataclass | LDU2 加密同步字段 |
| `PDU dict` | `dict` | scanner 输出的统一协议结果 |
