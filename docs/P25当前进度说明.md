# P25 Phase 1 当前进度说明

## 当前阶段

项目已经从最初的 P25 NID/NAC/DUID 元数据识别，推进到 **P25 Phase 1 完整链路解码完成**。

当前已实现 Frame Sync → NID → LDU1 LC / LDU2 ES / HDU HCW → Session 的全链路，可以从真实样本 `data/p25_1_78125.rawiq` 中解析出完整通话元数据：

```text
P25_LDU1  NAC=0x293  SRC=1  DST=1
P25_CALL  NAC=0x293  SRC=1  DST=1
```

该结果已用本机 `dsd-fme` 对同一前端音频交叉验证。`dsd-fme` 输出的 LCW payload 为：

```text
00 00 04 00 00 01 00 00 01
Group 1 Source 1
```

自研解析链路恢复出的字段与该 oracle 一致。

## 已完成功能

### 1. P25 基础识别链路

已实现 P25 Phase 1 Frame Sync 检测、符号恢复、NID 提取和 DUID 分类。

- `p25/constants.py` — FS 同步字、dibit 映射、DUID 名称、LDU/HDU 帧布局常量、LC/ES/HDU 交织位表
- `p25/sync.py` — 帧同步 NCC 检测（帧头锚点，非 DMR 中心同步）
- `p25/dsp.py` — 符号恢复（`recover_full_frame` 864 符号整帧）、NID bit 提取、LC/ES/HDU 去交织
- `p25/nid.py` — NID 解析（NAC 12bit + DUID 4bit），BCH 纠错已接入
- `p25/framing.py` — DUID → 帧类型分派（voice/control/terminator）
- `p25/decoder.py` — 统一 PDU 输出 + 稳定帧门控 + 会话组装集成

### 2. Galois 域 GF(2⁶) 运算引擎

`p25/galois.py` — GF(2⁶) 的 exp/log 表（本原多项式 x⁶+x+1，0x43）、乘法、求逆、多项式求值，为所有 RS 码提供代数基础。

### 3. 完整 FEC 纠错编码

`p25/fec.py` — 全部纯 Python 自实现，与项目 `okdmr` 风格一致、零新依赖：

| 编码 | 用途 | 能力 |
|------|------|------|
| Hamming(10,6,3) | LDU1 LC / LDU2 ES 内层 | 纠正 1-bit 错误 |
| RS(24,12,13) / GF(2⁶) | LDU1 Link Control 外层 | 纠正 6 符号错误 |
| RS(24,16,9) / GF(2⁶) | LDU2 加密同步外层 | 纠正 4 符号错误 |
| RS(36,20,17) / GF(2⁶) | HDU Header Code Word | 纠正 8 符号错误 |
| Golay(24,6,8) | HDU 内层 | 纠正 ≤3-bit 错误 |
| BCH(63,16,23) | NID 保护 | 纠正 ≤11-bit 错误（穷举译码） |
| CRC-16 CCITT | TSBK 校验 | 检错 |

所有 RS 码采用缩短 RS(63,k) 适配（DSD-FME 同款），支持 Berlekamp-Massey + Chien search 完整译码。

### 4. LDU1 Link Control 解码

`p25/lc_decode.py` + `p25/link_control.py` — LDU1 LCW 完整解析链路：

```text
LDU1 frame bits
  → LC_HEXBIT_POSITIONS 按 DSD-FME 节奏提取
  → Hamming(10,6,3) × 24
  → RS(24,12,13) / GF(2⁶)
  → 72-bit LCW
  → Group Voice / Unit-to-Unit 字段解析
```

已支持 LC format：
- **Group Voice Channel User** (LCO=0x00)：MFID / Emergency / TGID / SrcID
- **Unit-to-Unit Voice** (LCO=0x03)：MFID / SvcOpt / DstID / SrcID

真实样本输出：`TGID=1 SRC=1`

### 5. HDU Header Code Word 解码

`p25/hdu_decode.py` — HDU 帧解析：

```text
HDU frame bits
  → Golay(24,6,8) × 36 → 36 个 6-bit 符号
  → RS(36,20,17) / GF(2⁶)
  → 120-bit HCW: MI(72) + MFID(8) + ALGID(8) + KID(16) + TGID(16)
```

### 6. LDU2 Encryption Sync 解码

`p25/ldu2_decode.py` — LDU2 帧解析：

```text
LDU2 frame bits
  → ES_HEXBIT_POSITIONS 提取
  → Hamming(10,6,3) × 24
  → RS(24,16,9) / GF(2⁶)
  → 96-bit ES: MI(72) + ALGID(8) + KID(16)
```

### 7. P25 会话组装

`p25/session.py` — 将 HDU → LDU1/LDU2(交替) → TDU 序列组织为一次 `P25_CALL` 输出。

```text
P25_CALL  SRC=1  DST=1  NAC=0x293  duration_s=3.2  ldu_count=18
```

### 8. scanner 协议集成

`scanner.py` 已支持 DMR + P25 双协议分发：
- P25 去重 key 改为 `(nac, type, round(fs_start/帧长))`，避免所有 P25 帧因 `src=0,dst=0` 被错误折叠
- P25 稳定帧门控（跨帧 NAC 一致性 + BCH 校验），避免 DMR 样本被 P25 假阳性污染

当前 DMR smoke 输出保持干净：
```text
LC_HEADER   PROTO=DMR SRC=1 DST=1
LATE_ENTRY  PROTO=DMR SRC=1 DST=1
```

## 关键修正点

### LDU1 LC bit 位置

最初连续占位布局无法解析真实样本。已按 `dsd-fme` 的 LDU1 读取节奏修正——每 36 dibits 插入 status symbol，LC data/parity 分散在 IMBE2 到 IMBE7 之间。`LC_HEXBIT_POSITIONS` 由代码生成。

### Hamming(10,6,3)

使用 TIA/DSD 矩阵。策略：0-bit 直接通过、1-bit 修正、2-bit+ 保留原始 data bits 交给 RS 处理。

### RS(24,12,13) / RS(24,16,9) / RS(36,20,17)

全部采用 DSD-FME 同款缩短 RS(63,k) 适配：`parity + data + zero_padding → RS(63,51/55/47)`。

### LCW 字段顺序

LDU1 RS data symbol 恢复后，按 `hex_data[11]..hex_data[0]` 拼接成 72-bit LCW。

## 当前剩余工作

### 1. BCH NID 校验仍需完善

当前真实样本的 P25 NID 主要依赖跨帧 NAC 稳定性门控。BCH(63,16) 穷举译码在干净帧上可工作，真实样本上偶有失败。

### 2. TSBK / trunking control 暂未解析

目前 P25 控制类帧仍停留在 NID/DUID 层级。后续可以扩展 TSBK、TDULC、trunking grant、channel update。

### 3. LCW 格式覆盖有限

当前重点支持 `Group Voice Channel User` (LCO=0x00) 和 `Unit-to-Unit Voice` (LCO=0x03)。后续可按 `p25_lcw.c` 继续补其他 LCW format。

## 当前结论

P25 Phase 1 已达到与 DMR 对等的信令输出能力：

```text
Frame Sync → NID(BCH) → LDU1 LCW(Hamming+RS) → SRC/TGID → P25_CALL
                      → HDU HCW(Golay+RS) → MI/ALGID/KID/TGID
                      → LDU2 ES(Hamming+RS) → MI/ALGID/KID
```
