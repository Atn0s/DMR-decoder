# 多制式解调工程目标架构

本文描述工程后续重构的目标架构。目标不是一次性推翻现有实现，而是在保持 DMR、P25 Phase 1、dPMR 当前解码能力可回归的前提下，逐步把主线编排、协议知识、参数配置、输出格式和后续工程化边界拆清楚。

## 1. 设计目标

当前工程已经完成离线多制式模块化和主要工程化收口；剩余问题主要集中在 legacy 保留策略、realtime 重写、配置外部化和高风险算法参数治理：

- `scanner.py` 已瘦身为离线薄入口，输出层已拆到 `radio.output`。
- `dmr.offline.scan_file()` 已变为兼容 wrapper，需要保留一段迁移期。
- `core/` 名称像通用核心层，实际已经退化为 DMR 兼容 facade。
- realtime/宽带路径仍以 DMR 呼叫聚合为中心，非 DMR PDU 目前不会形成会话输出。
- 常用离线边界参数已进入 config 和参数参考；少量评分权重仍需结合样本指标谨慎迁移。
- 已引入 `radio.pdu.PDU` 边界类型，协议插件输出已归一化为完整 schema dict；协议内部 decoder 可继续渐进迁移。

目标架构遵守以下原则：

1. **主线只管信号流转，不包含协议知识。**
2. **协议模块自己声明前端、解码、后处理、去重、格式化和默认参数。**
3. **所有可调参数集中配置，带单位、说明、合法范围和调参影响。**
4. **离线和 realtime 尽量复用同一个窄带解码核心。**
5. **Python 实现和未来 C++ 实现通过同一接口替换，不改变上层业务代码。**
6. **`core/` 作为 legacy DMR facade 保留一段过渡期，但新代码不再依赖它。**

## 3. 主数据流

目标数据流如下：

```text
rawiq / SDR stream / wideband subband
    │
    ▼
IQBlock(samples, sample_rate, metadata)
    │
    ├─ 候选搜索：PSD / detector / explicit --fo
    │
    ▼
ChannelCandidate(fo_hz, source_sample_rate, metadata)
    │
    ▼
DDC + resample
    │
    ▼
NarrowbandIQ(samples, sample_rate, candidate)
    │
    ▼
ProtocolEngine.decode()
    │
    ├─ protocol.frontend()
    ├─ protocol.decode()
    ├─ protocol.postprocess()
    └─ protocol.dedup_key()
    │
    ▼
list[PDU]
    │
    ├─ formatter / JSON
    └─ session aggregator
```

离线入口和 realtime worker 的差异应该只在输入来源、窗口切片和候选来源上。进入 `NarrowbandIQ` 后，二者应尽量走同一套协议引擎。

## 4. 协议插件接口

当前 `ProtocolSpec` 只记录协议名和解码函数名。目标是把它升级成协议能力声明：

```python
@dataclass(frozen=True)
class ProtocolSpec:
    name: str
    aliases: tuple[str, ...]
    config_type: type
    default_config: object

    channel_bandwidth_hz: float
    target_sample_rate_hz: float
    symbol_rate_hz: float

    frontend: Callable[[np.ndarray, DecodeContext], np.ndarray]
    decode: Callable[[np.ndarray, DecodeContext], list[PDU]]
    postprocess: Callable[[list[PDU], DecodeContext], list[PDU]]
    dedup_key: Callable[[PDU], tuple]
    format_pdu: Callable[[PDU, FormatContext], str]
```

主线只通过 `ProtocolSpec` 调用协议，不再写协议分支：

```python
for spec in enabled_protocols:
    y = spec.frontend(narrowband.samples, context)
    pdus = spec.decode(y, context)
    pdus = spec.postprocess(pdus, context)
```

各协议职责：

| 协议 | frontend | decode | postprocess | dedup_key | format_pdu |
|------|----------|--------|-------------|-----------|------------|
| DMR | DMR/C4FM 4FSK 前端 | LC Header、Terminator、CSBK、Late Entry | 初期透传 | SRC/DST/type/频偏桶 | DMR 文本格式 |
| P25 | 复用通用 C4FM/4FSK 前端 | FS、NID、HDU、LDU1、LDU2、CALL | 初期透传 | NAC/type/frame bucket | P25 文本格式 |
| dPMR | dPMR 窄带前端 | FS1/FS2、CCH、Color Code | stable color filter | SRC/DST/color/frame bucket | dPMR 文本格式 |

这样新增协议时，应只新增协议包和 `plugin.py`，主线不需要继续增加 `if proto == ...`。

## 5. 第一步和第二步的具体含义

### 5.1 第一步：把协议能力登记到 ProtocolSpec

第一步不是重写解码算法，而是把现有函数按能力挂到 registry 上。行为应保持不变。

当前形态类似：

```python
ProtocolSpec("DMR", ("dmr",), "decode_dmr")
```

目标形态：

```python
ProtocolSpec(
    name="DMR",
    aliases=("dmr",),
    config_type=DMRConfig,
    default_config=DMRConfig(),
    channel_bandwidth_hz=12_500.0,
    target_sample_rate_hz=48_000.0,
    symbol_rate_hz=4_800.0,
    frontend=dmr_frontend,
    decode=dmr_decode,
    postprocess=dmr_postprocess,
    dedup_key=dmr_dedup_key,
    format_pdu=dmr_format_pdu,
)
```

落地时可以分批增加字段：

1. 先增加 `dedup_key` 和 `format_pdu`。
2. 再增加 `frontend` 和 `postprocess`。
3. 最后增加配置对象和更完整的协议元数据。

这样每一步都可以通过测试保证行为未变。

### 5.2 第二步：把 scanner.py 中的协议判断搬回协议模块

该步骤已经落地。`scanner.py` 当前只负责：

```python
print(protocols.format_pdu(pdu))
```

协议 formatter 由各协议插件声明：

```text
dmr/plugin.py   -> dmr_format_pdu()
p25/plugin.py   -> p25_format_pdu()
dpmr/plugin.py  -> dpmr_format_pdu()
```

协议去重 key 也已下沉到各自插件：

```text
dmr/plugin.py   -> dmr_dedup_key()
p25/plugin.py   -> p25_dedup_key()
dpmr/plugin.py  -> dpmr_dedup_key()
```

这一步完成后，`scanner.py` 不再知道 P25/dPMR/DMR 的输出字段细节。后续新增协议时，主入口不需要改格式化和去重逻辑。

## 6. 参数治理

参数治理是本次架构重构的核心部分。目标不是简单把常量搬到一个文件，而是让参数具备可发现、可解释、可校验、可覆盖的工程属性。

### 6.1 配置文件组织

建议新增：

```text
config/
  defaults.toml
  profiles/
    dmr.toml
    p25.toml
    dpmr.toml
    realtime.toml
    wideband.toml
```

`defaults.toml` 存放通用默认值，协议 profile 覆盖协议私有参数。命令行可以继续提供少量高频参数覆盖，例如 `--protocol`、`--fo`、`--config`、`--profile`。

### 6.2 配置对象

每个协议提供自己的配置 dataclass：

```python
@dataclass(frozen=True)
class DMRConfig:
    target_sample_rate_hz: float = 48_000.0
    symbol_rate_hz: float = 4_800.0
    frontend_cutoff_hz: float = 9_500.0
    nominal_deviation_hz: float = 1_944.0
    sync_threshold_voice: float = 0.68
    sync_threshold_data: float = 0.55
    burst_stride_samples: int = 2_880
```

```python
@dataclass(frozen=True)
class P25Config:
    target_sample_rate_hz: float = 48_000.0
    symbol_rate_hz: float = 4_800.0
    frontend_cutoff_hz: float = 9_500.0
    nominal_deviation_hz: float = 1_944.0
    sync_threshold: float = 0.62
```

```python
@dataclass(frozen=True)
class DPMRConfig:
    target_sample_rate_hz: float = 48_000.0
    symbol_rate_hz: float = 2_400.0
    frontend_cutoff_hz: float = 3_500.0
    nominal_deviation_hz: float = 1_944.0
    sync_threshold: float = 0.82
    sps_search_min: float = 19.0
    sps_search_max: float = 21.0
    sps_search_steps: int = 21
```

通用信号和 realtime 参数也应配置化：

```python
@dataclass(frozen=True)
class RadioConfig:
    wideband_default_sample_rate_hz: float = 2_500_000.0
    psd_peak_threshold_db: float = 15.0
    psd_nperseg: int = 4096
    candidate_min_distance_bins: int = 20
```

```python
@dataclass(frozen=True)
class RealtimeConfig:
    active_threshold_db: float = 15.0
    channel_grid_hz: float = 12_500.0
    close_hysteresis_windows: int = 3
    fo_bucket_hz: float = 5_000.0
    call_timeout_windows: int = 5
```

### 6.3 参数元数据

每个可调参数都应有说明元数据，至少包含：

| 字段 | 说明 |
|------|------|
| name | 参数名 |
| default | 默认值 |
| unit | 单位 |
| owner | 所属模块 |
| range | 合法范围或建议范围 |
| description | 参数用途 |
| tuning_effect | 调大/调小对误检、漏检、性能的影响 |
| user_tunable | 是否建议普通用户调整 |

这些元数据可以先写入 `docs/参数参考.md`，后续再由配置 schema 自动生成。

### 6.4 参数治理状态

已收敛到配置对象并在 [参数参考](参数参考.md) 中记录的参数：

| 当前位置 | 参数 | 目标归属 |
|----------|------|----------|
| `radio.pipeline.py` | PSD 门限、Welch 分段、峰间隔、采样率容差 | `RadioConfig` |
| `dmr.engine.py` / `dmr.plugin.py` | 同步门限、burst stride/count、去重窗口、频偏桶 | `DMRConfig` |
| `p25.decoder.py` / `p25.sync.py` / `p25.plugin.py` | sync 门限、sync 峰间隔、稳定 NAC 过滤、去重桶 | `P25Config` |
| `dpmr.decoder.py` / `dpmr.dsp.py` / `dpmr.plugin.py` | sync/SPS/phase 搜索、候选上限、stable color、去重桶 | `DPMRConfig` |
| `realtime.detector.py` / `realtime.aggregator.py` | detector/聚合器默认阈值 | `RealtimeConfig` |

暂缓迁移的只剩高风险算法评分参数，例如 FSK active gate、DMR burst phase sweep、P25 residual penalty、dPMR CCH/color score weights。它们已记录在 [参数参考](参数参考.md) 的“暂缓迁移的硬编码参数”中。

## 7. PDU 类型

当前已引入 `radio.pdu.PDU` 作为统一 PDU 边界类型，短期继续兼容 decoder 返回的 `dict`：

```python
@dataclass
class PDU:
    type: str
    protocol: str = "DMR"
    src: int | str = 0
    dst: int | str = 0
    ts: int | None = None
    flco: str = ""
    fid: str = ""
    extra: dict[str, Any] = field(default_factory=dict)
    raw_bits: Any = None
    meta: dict[str, Any] = field(default_factory=dict)
    fields: dict[str, Any] = field(default_factory=dict)
```

`meta` 用于保存非协议字段，例如：

- `fo_hz`，兼容 dict 输出为 `_fo_hz`
- `rf_hz`，兼容 dict 输出为 `_rf_hz`
- `window_id`，兼容 dict 输出为 `_window_id`
- candidate id
- decoder confidence

兼容期提供：

```python
PDU.from_dict(old_pdu)
PDU.to_dict(include_raw_bits=False)
pdu_get(pdu, "_fo_hz")
set_pdu_meta(pdu, "fo_hz", fo)
```

当前 `protocols.format_pdu()`、`protocols.dedup_key()`、`radio.output.write_json()`、`radio.pipeline`、`realtime.worker` 和 `realtime.aggregator` 已兼容 `PDU`。后续再逐步迁移 decoder 原生输出，而不是一次性大改所有调用点。

当前协议插件的 decode 输出已通过 `radio.pdu.pdus_to_standard_dicts()` 归一化为完整 schema dict；协议内部 decoder 仍可暂时返回历史 dict。

## 8. DMR 目标形态

DMR 离线编排已经收敛：`dmr.offline.py` 仅作为旧入口兼容 wrapper，真实离线路径走 `scanner.py` 和 `radio.pipeline`。

目标：

```text
dmr/constants.py   DMR 协议常量，逐步减少可调参数硬编码
dmr/config.py      DMRConfig
dmr/dsp.py         DMR 同步、burst 恢复、符号判决
dmr/engine.py      DMR burst 编排、Late Entry 恢复、decode 入口
dmr/decoder.py     LC Header、Terminator、CSBK、Late Entry 解码
dmr/plugin.py      DMR ProtocolSpec、formatter、dedup_key、postprocess
dmr/offline.py     旧离线入口兼容 wrapper；scan_file 转调统一 pipeline
```

`dmr.offline.py` 当前不再自己实现：

- PSD 搜索
- DDC
- 重采样
- 候选频偏循环
- 去重
- JSON 或打印

这些应全部进入统一 pipeline。

## 9. realtime 和宽带目标形态

realtime 当前仍是 DMR 聚合模型。目标是把 worker 和 aggregator 都协议化。

### 9.1 Worker

当前：

```text
DDC -> resample -> core.dsp.frontend -> protocols.decode_all
```

目标：

```text
DDC -> resample -> ProtocolEngine.decode_narrowband(enabled_protocols)
```

worker 已不再依赖旧的 `scanner.py` 内部 decode wrapper；后续目标是也不再 import `core.dsp.frontend`，而是复用离线窄带协议解码入口。

### 9.2 Aggregator

当前 `SessionAggregator.feed()` 直接忽略非 DMR PDU。目标是：

```text
protocol session policy:
  DMR  -> DMRCallAggregator
  P25  -> P25CallAggregator
  dPMR -> DPMRCallAggregator 或 PDU pass-through
```

可以先实现最小版本：

- DMR 保持现有 call record 行为。
- P25 使用 `P25_CALL` 输出形成 call record。
- dPMR 初期只做稳定 PDU 输出，不强制聚合为 call。

## 10. C++ 工程化适配

目标架构适合后续 C++ 重构，前提是接口按数据边界稳定下来，而不是按 Python 文件稳定。

适合优先替换为 C++ 的热点：

- DDC / 重采样
- 宽带 channelizer
- FSK discriminator / frontend
- NCC sync search
- symbol recovery
- FEC 解码
- 大批量候选解码循环

建议接口边界：

```text
IQ samples + sample_rate + frontend config
    -> discriminator / symbol stream

discriminator stream + protocol config
    -> PDU list

PDU list + session config
    -> call/session records
```

Python 层继续负责：

- CLI
- 配置加载和校验
- 文件 IO
- JSON 输出
- 测试编排
- 调试可视化

未来可以通过 `pybind11` 或独立动态库接入：

```python
if cpp_backend_available:
    spec.decode = cpp_p25_decode
else:
    spec.decode = python_p25_decode
```

只要 `ProtocolSpec`、`PDU` 和配置对象稳定，Python/C++ 实现可以并存，便于逐步替换性能热点。

## 11. 迁移状态

离线链路已完成：

1. 文档固定目标架构。
2. `ProtocolSpec` 挂载 frontend、decode、postprocess、dedup 和 formatter。
3. formatter、dedup、frontend、postprocess 下沉到协议插件。
4. 配置对象和 [参数参考](参数参考.md) 覆盖常用离线边界参数。
5. `radio.pipeline` 承接 PSD、DDC、重采样、协议解码、后处理和去重。
6. `radio.output` 承接格式化打印和 JSON 写出。
7. `dmr.offline.scan_file()` 改为兼容 wrapper。
8. `PDU` 边界类型已引入，协议插件输出已标准化。
9. realtime worker 已消除对 `scanner.py` 内部 decode wrapper 的依赖。

剩余非离线主链路工作：

1. 重新设计 realtime worker/aggregator，使非 DMR PDU 也能形成合理会话输出。
2. 为配置增加 TOML/CLI 覆盖和参数校验。
3. 根据样本指标迁移高风险算法评分参数。
4. 制定 `core/` facade 和历史调试脚本的保留/迁移策略。
5. 评估 C++ 后端替换点，优先替换性能热点，不改变 Python 上层接口。

## 12. 非目标

本次架构重构不应同时做以下事情：

- 不重写 DMR/P25/dPMR 的协议算法。
- 不立即删除 `core/`。
- 不立即把所有协议内部 decoder 改成原生 `PDU` 返回。
- 不立即把 realtime 改成完整三制式会话聚合。
- 不在没有测试保护的情况下改动同步阈值、FEC 行为或判决逻辑。

先把边界立住，再逐步迁移实现。
