# 离线多制式模块化结构

本文描述当前离线解码重构后的文件结构、入口关系和后续演进方向。

完整的数据流、协议内部中间产物和 realtime/宽带兼容路径见：

- [项目工作流程与数据流](项目工作流程与数据流.md)

## 目标

本次重构的目标是把 DMR 从原先根目录与 `core/` 的嵌套状态中剥离出来，使 DMR、P25、dPMR 三种制式形成对等模块，并保留一个统一离线扫描入口。

当前入口形态为 `1 + 3`：

- 主入口：`python scanner.py <iq_file>`
- DMR 单独入口：`python -m dmr <iq_file>`
- P25 单独入口：`python -m p25 <iq_file>`
- dPMR 单独入口：`python -m dpmr <iq_file>`

`realtime/` 与宽带信道化路径本次不做结构性重构，只做最小依赖改造：worker 直接调用协议 registry，不再依赖 `scanner.py` 内部函数。

## 当前文件结构

```text
common/
  __init__.py
  io.py              # 协议无关 IQ 文件读取、采样率文件名识别
  dsp.py             # 协议无关插值与窄带 FSK 前端

dmr/
  __init__.py
  __main__.py        # python -m dmr
  cli.py             # DMR 单独离线入口
  constants.py       # DMR 符号率、同步模板、Slot Data Type 等常量
  dsp.py             # DMR 前端、同步检测、突发恢复、四电平判决
  decode_flow.py     # DMR burst 编排、Late Entry 恢复、decode 入口
  link_layer.py      # DMR LC Header、Terminator、CSBK、Late Entry 解码
  plugin.py          # DMR ProtocolSpec、formatter、dedup、frontend、decode wrapper
  offline.py         # 旧 DMR 离线入口兼容 wrapper

p25/
  __main__.py        # python -m p25
  cli.py             # P25 单独离线入口
  decode_flow.py     # P25 同步候选遍历、整帧恢复、PDU 组装
  ...                # P25 FEC、同步、帧结构、会话组装等

dpmr/
  __main__.py        # python -m dpmr
  cli.py             # dPMR 单独离线入口
  decode_flow.py     # dPMR FS1/FS2 搜索、候选选择、PDU 组装
  link_layer.py      # dPMR CCH、Color Code、payload 解析
  dsp.py             # dPMR 前端与符号恢复
  ...                # dPMR CCH、Color Code、会话组装等

core/
  burst_type.py      # 兼容层，转发到 dmr.constants
  dsp.py             # 兼容层，转发到 dmr.dsp
  decoder.py         # 兼容层，转发到 dmr.link_layer

radio/
  protocol.py        # ProtocolSpec 定义
  registry.py        # 协议 registry、协议名归一化、统一分发、PDU 格式化和去重 key
  pipeline.py        # 离线 IQ 编排：PSD、DDC、重采样、协议解码、后处理、去重
  pdu.py             # PDU 边界类型、dict 兼容转换和元数据写入
  output.py          # 离线结果格式化打印和 JSON 写出

scanner.py           # 统一离线薄入口
```

## 调用路径

主入口 `scanner.py` 只负责文件和 CLI 层编排：

1. `common.io.read_rawiq()` 读取 interleaved int16 IQ。
2. `detect_sample_rate()` 从文件名识别采样率，例如 `dmr_1_78125.rawiq`。
3. 调用 `radio.pipeline.scan_iq()`。
4. 通过 `radio.output.print_results()` 打印结果，或通过 `radio.output.write_json()` 写 JSON。

`radio.pipeline.py` 负责协议无关的离线 IQ 编排：

1. 默认把输入视为已居中的 baseband IQ，直接重采样到 48 kHz。
2. 如果调用方传入 `freq_list`，对每个指定频偏执行 DDC、重采样。
3. 如果调用方显式启用 blind search，先做 Welch PSD 候选搜索，再对每个候选执行 DDC、重采样。
4. 调用 `radio.registry.decode_iq_enabled()`，由协议 `ProtocolSpec` 生成各自前端并执行解码。
5. 执行协议后处理和跨候选去重。

`radio.registry` 当前通过各协议 `plugin.SPEC` 组成 `ProtocolSpec` registry：

- `DMR` -> `dmr.plugin.decode(y)` -> `dmr.decode_flow.decode_dmr_flow(y)`
- `P25` -> `p25.plugin.decode(y)` -> `p25.decode_flow.decode(y)`
- `dPMR` -> `dpmr.plugin.decode(y)` -> `dpmr.decode_flow.decode(y)`

协议名统一由 `radio.registry.normalize_protocol_names()` 处理，支持 `dmr`、`p25`、`dpmr` 等命令行别名。跨候选去重 key 由 `radio.registry.dedup_key()` 生成，`scanner.py` 不再直接持有 P25/dPMR/DMR 的去重字段细节。

单独入口中，DMR、P25 和 dPMR 都复用 `scanner.scan_file(..., protocol_names=[...])`，这样可以共享主入口的 IQ 读取、重采样、打印和 JSON 行为。`dmr.offline.scan_file()` 仅作为旧调用兼容 wrapper 保留。

## 兼容策略

为了不破坏现有测试、调试脚本和实时路径，`core/` 暂时保留为兼容层：

- 旧代码 `from core.dsp import frontend` 仍可运行。
- 新代码应直接使用 `dmr.dsp`、`dmr.link_layer`、`dmr.constants`。
- `scanner.py` 不再暴露旧的 decode、DMR decode 和候选处理内部兼容函数。
- `realtime/worker.py` 当前直接调用 `radio.registry.decode_iq()`，避免继续绑定离线主入口内部 API。

这意味着本次重构完成后，DMR 已经成为独立模块，离线入口和协议核心之间也通过 `radio.pipeline` / `protocols` 解耦；历史 `core.*` import 不会立即失效。

## 新增制式的接入方式

后续新增制式时，建议按以下最小接口组织：

```text
new_protocol/
  __init__.py
  __main__.py
  cli.py
  constants.py
  dsp.py
  decode_flow.py
  link_layer.py      # 如该制式需要单独的链路层字段解析
  plugin.py
```

然后在 `new_protocol/plugin.py` 中声明 `ProtocolSpec`：

```python
SPEC = ProtocolSpec(
    name="NEW",
    aliases=("new",),
    config=NewProtocolConfig(),
    frontend_key="new",
    frontend=new_frontend,
    decode=new_decode,
    postprocess=postprocess_identity,
    dedup_key=new_dedup_key,
    formatter=new_format_pdu,
)
```

最后在 `radio/registry.py` 的 registry 中导入该 `SPEC`。新增制式不需要改 `scanner.py`。

如果新协议需要不同前端，应通过 `ProtocolSpec.frontend` 和 `frontend_key` 描述，`radio.registry.decode_iq()` 会按 key 复用同类前端输出。

## 剩余实现内容

离线主链路的结构性重构已经完成。剩余内容主要是兼容期和非离线主链路工作：

1. **协议内部原生 PDU**
   已引入 `radio.pdu.PDU` 作为边界类型，并让协议插件输出归一化为完整 schema dict。后续可以逐步让各协议内部 decoder 原生返回 `PDU`。

2. **配置外部化**
   常用离线参数已进入 config dataclass 和 [参数参考](参数参考.md)。后续可增加 TOML/CLI 覆盖、参数校验和 profile 管理。

3. **清理历史脚本**
   `dmr_pipeline_v2.py`、`dmr_symbol_engine.py`、`late_entry.py` 仍保留旧路径。后续可迁入 `debug/dmr/` 或 `dmr/legacy/`，并逐步替换 `core.*` import。

4. **实时路径统一**
   当前 `realtime/worker.py` 已不再依赖旧的 `scanner.py` 内部 decode wrapper，但仍保留旧的 DMR 聚合模型和 `core.dsp.frontend` 兼容入口。后续可基于离线 pipeline 重新实现实时链路。

5. **宽带信道化协议扩展**
   宽带路径目前复用离线窄带解码核心。后续新增更多制式时，需要为每种制式声明候选带宽、默认前端、采样率、窗口长度和去重策略。
