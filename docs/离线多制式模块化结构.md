# 离线多制式模块化结构

本文描述当前离线解码重构后的文件结构、入口关系和后续演进方向。

完整的数据流、协议内部中间产物和 realtime/宽带兼容路径见：

- [项目工作流程与数据流](项目工作流程与数据流.md)

## 目标

本次重构的目标是把 DMR 从原先根目录与 `core/` 的嵌套状态中剥离出来，使 DMR、P25、dPMR 三种制式形成对等模块，并保留一个统一离线扫描入口。

当前入口形态为 `1 + 3`：

- 主入口：`python scanner.py <iq_file>`
- DMR 单独入口：`python -m dmr <iq_file>`
- P25 单独入口：`python -m p25 <iq_file>`
- dPMR 单独入口：`python -m dpmr <iq_file>`

`realtime/` 与宽带信道化路径本次不做结构性重构，只通过兼容 wrapper 继续复用离线协议分发。

## 当前文件结构

```text
common/
  __init__.py
  io.py              # 协议无关 IQ 文件读取、采样率文件名识别
  dsp.py             # 协议无关插值与窄带 FSK 前端

dmr/
  __init__.py
  __main__.py        # python -m dmr
  cli.py             # DMR 单独离线入口
  constants.py       # DMR 符号率、同步模板、Slot Data Type 等常量
  dsp.py             # DMR 前端、同步检测、突发恢复、四电平判决
  decoder.py         # DMR LC Header、Terminator、CSBK、Late Entry 解码
  offline.py         # DMR 离线扫描与原 _decode_dmr_loop 实现

p25/
  __main__.py        # python -m p25
  cli.py             # P25 单独离线入口
  decoder.py         # P25 协议解码主 API
  ...                # P25 FEC、同步、帧结构、会话组装等

dpmr/
  __main__.py        # python -m dpmr
  cli.py             # dPMR 单独离线入口
  decoder.py         # dPMR 协议解码主 API
  dsp.py             # dPMR 前端与符号恢复
  ...                # dPMR CCH、Color Code、会话组装等

core/
  burst_type.py      # 兼容层，转发到 dmr.constants
  dsp.py             # 兼容层，转发到 dmr.dsp
  decoder.py         # 兼容层，转发到 dmr.decoder

protocols.py         # 协议 registry、协议名归一化、离线分发、PDU 去重 key
scanner.py           # 统一离线扫描入口
```

## 调用路径

主入口 `scanner.py` 负责协议无关的离线编排：

1. `common.io.read_rawiq()` 读取 interleaved int16 IQ。
2. `detect_sample_rate()` 从文件名识别采样率，例如 `dmr_1_78125.rawiq`。
3. 窄带文件直接重采样到 48 kHz，宽带文件先做 Welch PSD 候选搜索。
4. 对每个候选频偏执行 DDC、重采样和协议前端。
5. 调用 `protocols.py` 分发到启用的协议模块。
6. 执行跨候选去重、dPMR 稳定性过滤、打印和 JSON 输出。

`protocols.py` 当前通过 `ProtocolSpec` registry 注册三个协议：

- `DMR` -> `decode_dmr(y)` -> `dmr.offline._decode_dmr_loop(y)`
- `P25` -> `decode_p25(y)` -> `p25.decoder.decode(y)`
- `dPMR` -> `decode_dpmr(y)` -> `dpmr.decoder.decode(y)`

协议名统一由 `protocols.normalize_protocol_names()` 处理，支持 `dmr`、`p25`、`dpmr` 等命令行别名。跨候选去重 key 由 `protocols.dedup_key()` 生成，`scanner.py` 不再直接持有 P25/dPMR/DMR 的去重字段细节。

单独入口中，DMR 使用 `dmr.offline.scan_file()`；P25 和 dPMR 复用 `scanner.scan_file(..., protocol_names=[...])`，这样可以共享主入口的 IQ 读取、重采样、打印和 JSON 行为。

## 兼容策略

为了不破坏现有测试、调试脚本和实时路径，`core/` 暂时保留为兼容层：

- 旧代码 `from core.dsp import frontend` 仍可运行。
- 新代码应直接使用 `dmr.dsp`、`dmr.decoder`、`dmr.constants`。
- `scanner._decode_loop()` 仍保留，供 `realtime/worker.py` 继续调用。
- `scanner._decode_dmr_loop()` 仍保留，内部转调 `dmr.offline._decode_dmr_loop()`。

这意味着本次重构完成后，DMR 已经成为独立模块，但历史 import 不会立即失效。

## 新增制式的接入方式

后续新增制式时，建议按以下最小接口组织：

```text
new_protocol/
  __init__.py
  __main__.py
  cli.py
  constants.py
  dsp.py
  decoder.py
```

然后在 `protocols.py` 中增加：

```python
from new_protocol.decoder import decode as decode_new_protocol
```

并在 `scanner.SUPPORTED_PROTOCOLS` 和 `_decode_protocol_frontends()` 中注册该协议。

如果新协议需要不同前端，应在 `scanner.py` 中为该协议生成独立 frontend 输出，或把 frontend 调用下沉到协议模块。

## 后续实现内容

本次重构完成的是离线模块化第一阶段。后续建议按以下顺序继续：

1. **统一 PDU 类型**
   当前输出仍是 dict，字段约定为 `protocol/type/src/dst/ts/flco/fid/extra/raw_bits`。后续可引入 dataclass 或 TypedDict，减少聚合、打印、JSON 输出中的隐式字段判断。

2. **格式化继续下沉**
   当前 P25/dPMR/DMR 的打印仍集中在 `scanner.py`。后续可把 `format_pdu()` 放入各协议模块，或扩展 `ProtocolSpec` 支持 formatter。

3. **清理历史脚本**
   `dmr_pipeline_v2.py`、`dmr_symbol_engine.py`、`late_entry.py` 仍保留旧路径。后续可迁入 `debug/dmr/` 或 `dmr/legacy/`，并逐步替换 `core.*` import。

4. **实时路径统一**
   当前 `realtime/worker.py` 通过 `scanner._decode_loop()` 兼容运行。后续可让 realtime 和 offline 共同依赖协议 registry，避免内部函数调用。

5. **宽带信道化协议扩展**
   宽带路径目前复用离线窄带解码核心。后续新增更多制式时，需要为每种制式声明候选带宽、默认前端、采样率、窗口长度和去重策略。
