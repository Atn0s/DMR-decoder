# 项目工作流程与数据流

本文描述当前工程在重构后的整体工作流程、数据流动路径和主要中间产物。重点覆盖离线多制式扫描主链路，同时说明 DMR、P25、dPMR 三个协议模块内部如何消费前端数据，以及 realtime/宽带部分如何复用离线核心。

## 1. 入口总览

当前工程有 `1 + 3` 个离线入口：

| 入口 | 命令 | 作用 |
|------|------|------|
| 统一主入口 | `python scanner.py <iq_file>` | 对输入 IQ 同时运行 DMR、P25、dPMR 检测与解码 |
| DMR 单独入口 | `python -m dmr <iq_file>` | 只运行 DMR 离线解码链路 |
| P25 单独入口 | `python -m p25 <iq_file>` | 通过主 scanner 只启用 P25 |
| dPMR 单独入口 | `python -m dpmr <iq_file>` | 通过主 scanner 只启用 dPMR |

主入口支持协议过滤：

```bash
python scanner.py data/dmr_1_78125.rawiq --protocol dmr
python scanner.py data/p25_1_78125.rawiq --protocol p25
python scanner.py data/dpmr_1_48000.rawiq --protocol dpmr
```

## 2. 核心模块职责

| 模块 | 职责 |
|------|------|
| `common/io.py` | 读取 interleaved int16 IQ，识别文件名中的采样率 |
| `common/dsp.py` | 协议无关插值、窄带 FSK 前端 |
| `radio/pipeline.py` | 离线 IQ 编排：PSD 候选搜索、DDC、重采样、协议解码、后处理和去重 |
| `radio/protocol.py` | `ProtocolSpec` 定义 |
| `radio/registry.py` | 协议 registry、协议名归一化、统一分发、协议相关格式化和去重 key |
| `radio/pdu.py` | PDU 边界类型、dict 兼容转换和元数据写入 |
| `radio/output.py` | 离线输出层：格式化打印、JSON 准备和 JSON 写出 |
| `scanner.py` | 离线薄入口：CLI、读文件、采样率识别、调用 pipeline、打印和 JSON 输出 |
| `dmr/` | DMR 常量、DSP、突发恢复、LC/CSBK/Late Entry 解码、DMR 单入口 |
| `p25/` | P25 frame sync、NID、HDU/LDU/LC、session 组装 |
| `dpmr/` | dPMR FS1/FS2 同步、符号恢复、CCH、Color Code、session 组装 |
| `core/` | 兼容层，旧 import 转发到 `dmr.*` |
| `realtime/` | 实时/宽带信道化链路，当前最小复用 `radio.registry.decode_iq()`，后续可按离线 pipeline 重写 |

## 3. 离线主链路数据流

统一入口 `scanner.scan_file()` 是离线主链路的文件/CLI 入口；实际 IQ 编排在 `radio.pipeline.scan_iq()`。整体数据流如下：

```text
rawiq/bvsp-derived IQ file
    │
    ▼
common.io.read_rawiq()
    │  np.ndarray[complex]，按 IQ dtype 自动归一化
    ▼
detect_sample_rate()
    │  从文件名识别 fs，例如 *_78125.rawiq -> 78125
    ▼
radio.pipeline.scan_iq()
    │
    ├─ 默认 baseband 模式
    │      │
    │      ▼
    │   process_baseband() -> resample_poly -> 48 kHz
    │
    ├─ freq_list 指定频偏
    │      │
    │      ▼
    │   process_candidate(): DDC -> resample_poly -> 48 kHz
    │
    └─ blind_search=True
           │
           ▼
        psd_blind_search() -> 候选频偏 fos
           │
           ▼
        process_candidate(): DDC -> resample_poly -> 48 kHz
    │
    ▼
radio.registry.decode_iq_enabled()
    │
    ├─ spec.frontend_key 复用前端输出
    ├─ DMR/P25: dmr.dsp.frontend()
    └─ dPMR: dpmr.dsp.frontend_dpmr()
    │
    ▼
spec.decode()
    │
    ├─ DMR decoder
    ├─ P25 decoder
    └─ dPMR decoder
    │
    ▼
PDU list
    │
    ├─ radio.registry.postprocess_pdus_enabled()
    └─ radio.registry.deduplicate_pdus()
    │
    ▼
打印结果 / 可选 JSON
```

### 3.1 `scanner.py` 函数级工作流程

`scanner.py` 是离线薄入口。它只负责 CLI、文件读取、采样率识别、调用统一 pipeline、打印和 JSON 输出；PSD 搜索、DDC、重采样、协议前端、协议解码、后处理和去重都在 `radio.pipeline` / `radio.registry` / 各协议 `plugin.py` 中完成。

整体调用关系：

```text
main()
  └─ scan_file()
       ├─ read_rawiq()
       ├─ detect_sample_rate()
       ├─ radio.pipeline.scan_iq()
       │    ├─ psd_blind_search() / process_candidate() / process_baseband()
       │    ├─ radio.registry.decode_iq_enabled()
       │    │    ├─ spec.frontend()
       │    │    └─ spec.decode()
       │    ├─ radio.registry.postprocess_pdus_enabled()
       │    └─ radio.registry.deduplicate_pdus()
       ├─ radio.output.print_results()
       └─ radio.output.write_json()     # 可选
```

#### 3.1.1 入口与输出

| 函数 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `main(argv=None)` | `argv: list[str] | None` | `int` | CLI 入口。解析目标文件、`--protocol`、`--fo`、`--fs`、`--blind-search`、IQ dtype、`--json`，逐文件调用 `scan_file()`。 |
| `scan_file(path, freq_list=None, output_json=None, protocol_names=None, sample_rate=None, blind_search=False, iq_dtype="int16")` | `path: str`；可选采样率、频偏列表、盲搜开关、JSON 输出路径、协议名列表、IQ dtype | `list[dict]` PDU | 离线扫描主函数。读取 IQ 和采样率后转调 `radio.pipeline.scan_iq()`。 |
| `detect_sample_rate(path)` | `path: str` | `int | None` | 从文件名中解析采样率，例如 `data/dmr_1_78125.rawiq -> 78125`。解析失败返回 `None`。 |
| `radio.output.print_results(pdus)` | `list[dict | PDU]` | `None` | 逐条调用 `radio.registry.format_pdu()` 输出文本。 |
| `radio.output.write_json(pdus, path)` | PDU 列表、输出路径 | `None` | 通过 `pdu_to_dict(..., include_raw_bits=False)` 写 JSON 文件。 |

`scan_file()` 本身不再持有协议前端、候选搜索、去重或格式化细节；新增制式不需要修改 `scanner.py`。

#### 3.1.2 离线 pipeline

`radio.pipeline.scan_iq()` 的输入分支：

```text
freq_list is not None
  -> 使用用户指定频偏，逐个调用 process_candidate()

blind_search=True
  -> 显式宽带/多候选搜索
  -> psd_blind_search() 找候选频偏
  -> 每个候选调用 process_candidate()

否则
  -> 视为已居中的 baseband IQ
  -> process_baseband()
```

`process_candidate()` 完成 DDC、重采样和协议解码，并用 `set_pdu_meta(pdu, "fo_hz", fo)` 标记候选频偏；`PDU.to_dict()` 会继续兼容输出 `_fo_hz`。

#### 3.1.3 协议分发

协议分发实际发生在 `radio.registry`：

```text
radio.registry.decode_iq_enabled(iq_dec, enabled_protocols=names, sample_rate=48_000)
  -> registry 顺序遍历 DMR / P25 / dPMR
  -> 按 spec.frontend_key 复用协议前端输出
  -> 调用 spec.decode()
```

后处理和去重也通过 `ProtocolSpec` 分发：

```text
radio.registry.postprocess_pdus_enabled()
  -> dPMR stable color filter
  -> DMR/P25 当前透传

radio.registry.deduplicate_pdus()
  -> spec.dedup_key()
```

#### 3.1.4 `scanner.py` 中间产物汇总

| 阶段 | 变量 | 类型 | 生命周期 | 是否输出到文件 |
|------|------|------|----------|----------------|
| 读文件 | `iq` | complex ndarray | `scan_file()` 内 | 否 |
| 采样率判断 | `fs` | `int | None` | `scan_file()` 内 | 否 |
| 宽带粗扫 | `fos` | `list[float]` | `radio.pipeline.scan_iq()` 宽带分支内 | 否 |
| DDC | `iq_shifted` | complex ndarray | `radio.pipeline.process_candidate()` 内 | 否 |
| 重采样 | `iq_dec` | complex ndarray | `radio.pipeline.process_candidate()` / `process_baseband()` 内 | 否 |
| DMR/P25 前端 | `y` | float ndarray | 单次候选/窄带处理内 | 否 |
| dPMR 前端 | `y_dpmr` | float ndarray | 单次候选/窄带处理内 | 否 |
| 协议输出 | `all_pdus` | `list[dict | PDU]` | `radio.pipeline.scan_iq()` 内 | 可经 JSON 输出 |
| 去重输出 | `unique` | `list[dict | PDU]` | `scan_file()` 返回 | 可经 JSON 输出 |
| 频偏标记 | `_fo_hz` | `float` | 宽带候选 PDU 内 | JSON 保留 |
| 原始比特 | `raw_bits` | `bytes` | PDU 内 | JSON 输出时删除 |

## 4. 主要数据对象

| 名称 | 类型 | 产生位置 | 含义 |
|------|------|----------|------|
| `iq` | `np.ndarray` complex | `common.io.read_rawiq()` | 原始复数 IQ，来自 int16 I/Q 交错文件 |
| `fs` | `int | None` | `scanner.detect_sample_rate()` | 输入采样率，通常从文件名推断 |
| `fos` | `list[float]` | `radio.pipeline.psd_blind_search()` | 宽带粗扫得到的候选频偏 |
| `iq_shifted` | `np.ndarray` complex | `radio.pipeline.process_candidate()` | 已按候选频偏 DDC 到零中频附近的 IQ |
| `iq_dec` | `np.ndarray` complex | `resample_poly` | 重采样到 48 kHz 的窄带 IQ |
| `y` | `np.ndarray` float | `dmr.dsp.frontend()` | FM 鉴频、去 DC、归一化后的符号尺度样点流 |
| `y_dpmr` | `np.ndarray` float | `dpmr.dsp.frontend_dpmr()` | dPMR 专用窄带前端输出 |
| `PDU` | `dict | radio.pdu.PDU` | 各协议 decoder / PDU 边界层 | 统一输出事件，包含协议、类型、源/目的、附加字段和原始 bits |
| `CallRecord` | dataclass | `realtime/aggregator.py` | realtime 聚合后的呼叫记录 |

各协议内部 decoder 当前主要输出 dict；协议插件边界会通过 `radio.pdu.pdus_to_standard_dicts()` 归一化为完整 schema dict。跨模块边界也可用 `radio.pdu.PDU` 包装并保留 dict 读写兼容。约定字段如下：

```python
{
    "protocol": "DMR" | "P25" | "dPMR",
    "type": "...",
    "src": ...,
    "dst": ...,
    "ts": 0,
    "flco": "...",
    "fid": "...",
    "extra": {...},
    "raw_bits": b"...",
}
```

JSON 输出会去掉 `raw_bits`，避免二进制字段污染结果文件。

## 5. DMR 模块内部流程

DMR 的离线协议核心位于 `dmr/engine.py`、`dmr/dsp.py`、`dmr/decoder.py`；`dmr/offline.py` 仅作为旧 DMR-only 调用兼容 facade。

```text
y
 │
 ▼
dmr.dsp.find_sync_positions()
 │  输出 [(center_sample, polarity, sync_type), ...]
 │
 ├─ DATA sync: DATA_MS / DATA_BS
 │      │
 │      ▼
 │   dmr.dsp.recover_burst()
 │      │  132 symbols，已按同步码仿射校准
 │      ▼
 │   dmr.decoder.decode_burst()
 │      │
 │      ├─ Slot Type Golay(20,8,7)
 │      ├─ BPTC(196,96)
 │      ├─ RS(12,9,4)
 │      └─ FullLinkControl / CSBK
 │
 └─ VOICE sync: MS_VOICE / BS_VOICE
        │
        ▼
     _lock_voice_phase()
        │
        ▼
     _recover_stepped_burst()，按 60 ms stride 恢复 A-F burst
        │
        ▼
     LateEntryCollector.feed()
        │
        ├─ Embedded Signalling
        ├─ LCSS fragment state machine
        ├─ VBPTC(128,72)
        └─ FiveBitChecksum
```

DMR 主要中间产物：

| 中间产物 | 含义 |
|----------|------|
| `positions` | DMR 同步位置、极性和同步类型 |
| `symbols` | 一个 132-symbol burst 的四电平符号 |
| `ba264` | 一个 DMR burst 对应的 264-bit bitarray |
| `slot_bits` | Slot Type 20 bit，用于 Golay 和 data type 判断 |
| `info196` | BPTC 输入 196 bit |
| `decoded[0:96]` | BPTC 输出，FLC/CSBK 的协议负载 |
| `LateEntryCollector._frags` | 语音突发中的 4 个 32-bit embedded LC fragment |

## 6. P25 模块内部流程

P25 的主入口是 `p25.decoder.decode(y)`。

```text
y
 │
 ▼
p25.sync.find_frame_sync()
 │  输出 P25SyncCandidate(fs_start, polarity, ncc)
 │
 ▼
p25.dsp.recover_symbols_from_fs()
 │  恢复 FS + NID 符号
 │
 ▼
slice_symbols_to_bits() + extract_nid_bits()
 │
 ▼
p25.nid.decode_nid()
 │  NAC / DUID / BCH 状态
 │
 ▼
p25.framing.frame_info_from_nid()
 │
 ├─ HDU: recover_full_frame() -> decode_hdu_hcw()
 ├─ LDU1: recover_full_frame() -> decode_ldu1_lc()
 └─ LDU2: recover_full_frame() -> decode_ldu2_es()
 │
 ▼
P25SessionAssembler.feed()
 │
 ▼
P25_NID / P25_HDU / P25_LDU1 / P25_LDU2 / P25_CALL PDU
```

P25 主要中间产物：

| 中间产物 | 含义 |
|----------|------|
| `P25SyncCandidate` | 帧同步起点、极性和 NCC 分数 |
| `nid_bits` | P25 NID BCH 输入位 |
| `P25NID` | NAC、DUID、BCH 校验/纠错状态 |
| `P25FrameInfo` | DUID 解释后的帧类型、是否语音/控制/终止 |
| `LinkControl` | LDU1 中恢复出的 SRC、DST/TGID、MFID、LCO |
| `HeaderCodeWord` | HDU 中恢复出的 MI、ALGID、KID、TGID |
| `EncryptionSync` | LDU2 中恢复出的加密同步字段 |
| `P25SessionAssembler` 状态 | 跨 HDU/LDU/TDU 组装一次通话 |

## 7. dPMR 模块内部流程

dPMR 的主入口是 `dpmr.decoder.decode(y_dpmr)`。dPMR 使用自己的窄带前端和符号恢复参数。

```text
y_dpmr
 │
 ├─ Header path
 │    │
 │    ▼
 │  find_fs1_sync()
 │    │
 │    ▼
 │  recover_frame_symbol_candidates()
 │    │
 │    ├─ decode_cch()
 │    └─ get_color_code()
 │
 └─ Voice path
      │
      ▼
    find_fs2_sync()
      │
      ▼
    recover_voice_fs2_symbol_candidates()
      │
      ▼
    split_voice_fs2()
      │
      ├─ CCH0
      ├─ Color Code
      └─ CCH1
      │
      ▼
    DPMRSessionAssembler.feed()
```

dPMR 主要中间产物：

| 中间产物 | 含义 |
|----------|------|
| `DPMRSyncCandidate` | FS1/FS2 同步起点、极性、NCC、同步类型 |
| `DPMRSymbolCandidate` | 符号序列、残差、SPS、相位、判决质量 |
| `CCHRecord` | CCH 解码结果，包括帧号、ID half、CRC/Hamming 状态 |
| `color_code` | dPMR color code |
| `quality` | CRC/Hamming/帧号组合得到的置信度摘要 |
| `DPMRSessionAssembler` 状态 | 根据 CCH 片段组装 SRC/DST |

`radio.pipeline.scan_iq()` 在协议解码后调用 `radio.registry.postprocess_pdus_enabled()`；dPMR 的 `ProtocolSpec.postprocess` 会执行 stable color filter，根据稳定 color code 和质量字段过滤 dPMR 假阳性。

## 8. 去重和输出

协议去重逻辑在 `radio.registry`：

| 协议 | 去重 key |
|------|----------|
| DMR | `(protocol, src, dst, type, fo_bucket)` |
| P25 | `(P25, nac, type, frame_bucket)` |
| dPMR | `(dPMR, src, dst, color_code, frame_bucket)` |

其中：

- `fo_bucket = round(_fo_hz / 5000) * 5000`
- P25 `frame_bucket = round(fs_start / 8640)`
- dPMR `frame_bucket = round(fs_start / 3840)`

输出路径：

```text
PDU list
 │
 ├─ radio.output.print_results() -> stdout
 │
 └─ radio.output.write_json() -> JSON 文件
        │
        └─ raw_bits 字段被剔除
```

## 9. realtime 与宽带信道化数据流

本次模块化重构没有重排 `realtime/`，但实时 worker 已不再依赖 `scanner.py` 的内部函数，当前最小复用协议 registry。

窄带/宽带实时数据流：

```text
File/SDR IQ source
    │
    ▼
realtime.iq_source / wideband_source
    │
    ▼
RingBuffer
    │
    ▼
固定长度重叠 window
    │
    ▼
Detector / Channelizer
    │  输出活跃候选信道和频偏
    ▼
realtime.worker.decode_window()
    │
    ├─ DDC
    ├─ resample_poly -> 48 kHz
    ├─ dmr.dsp/core.dsp frontend 兼容入口
    └─ radio.registry.decode_iq()
    │
    ▼
PDU list，附加 _fo_hz / _window_id
    │
    ▼
realtime.aggregator.SessionAggregator
    │
    ▼
CallRecord / on_call 回调
```

宽带信道化路径 `realtime/wideband_scanner.py` 会先把宽带 IQ 分成多个子带，再对子带窗口复用同样的 `decode_window()` 逻辑。

## 10. 文件级中间产物

| 路径 | 类型 | 说明 |
|------|------|------|
| `data/*.rawiq` | 输入样本 | int16 I/Q 交错原始样本 |
| `DMR_signal/*.bvsp` | 输入样本 | USRP/BVSP 宽带采集，需按 BVSP 头解析 |
| `output/*.png` | 调试图 | 历史调试脚本生成的 PSD、同步、星座/质量图 |
| `output/*.rawiq` | 合成/场景数据 | 宽带 demo 或合成器生成的中间场景 |
| `output/*.gif` | 可视化结果 | 宽带 demo 动画 |
| `--json <path>` | 解码结果 | scanner/单协议入口输出的 PDU JSON，去除 `raw_bits` |

多数协议中间产物只存在于内存中；除非运行 debug/visualize 脚本，否则不会自动写入 `output/`。

## 11. 推荐调试顺序

当某个样本解不出来时，建议按以下顺序定位：

1. 确认输入采样率：文件名是否包含 `_<fs>.rawiq`，否则 scanner 会按宽带默认路径处理。
2. 窄带样本先跑单协议入口，减少其他协议假阳性干扰。
3. 宽带样本先看 Welch 候选频偏，必要时用 `--fo` 手动指定候选。
4. DMR 优先检查同步位置、Golay、BPTC/RS。
5. P25 优先检查 Frame Sync、NID BCH、NAC 稳定性。
6. dPMR 优先检查 FS1/FS2、CCH CRC/Hamming、color code 稳定性。
7. realtime 问题先复现到离线 `scanner.py`，再检查 window 长度、step、候选频偏和 aggregator。
