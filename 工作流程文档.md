## 3. 流水线 A：LC Header 解码

### 3.1 整体目标

从 2.5 MHz 宽带 IQ 文件中**盲搜**所有 DMR 信号，对每一路执行解调，定位 Voice LC Header 突发，解码出呼叫的源地址、目标地址、FLCO、FID 等链路控制信息。

### 3.2 各阶段详述

---

#### Stage 1: 宽带粗扫

**目标**：在 2.5 MHz 频谱中检测所有活动的信号载波。

**输入**：
- `synthesized_wideband_2.5MHz.rawiq` — int16 交错 IQ，等效 `(I + jQ) / 32768`

**处理流程**：

```
IQ 数据 (Fs=2.5MHz)
    │
    ▼
scipy.signal.welch(nperseg=4096, return_onesided=False)
    │  韦尔奇法：分段加窗 → FFT → 功率平均，降低方差
    ▼
PSD (dB/Hz)
    │
    ▼
中值噪声基底估计: nf = median(PSD_dB)
检测门限: nf + 15 dB
    │
    ▼
scipy.signal.find_peaks(height=nf+15, distance=20)
    │  distance=20 → 最小频率间隔 = 20×bin ≈ 12.2 kHz，略小于 DMR 12.5kHz 信道间距
    ▼
候选频率列表: [f_offset_1, f_offset_2, ...]
```

**输出**：
- `peaks` — 候选信号的频率索引
- `f_w[peaks]` — 各候选的绝对频率偏移 (Hz)

**为什么这样设计**：
- Welch 法比简单 FFT 的方差更低，能在 20dB 噪声下可靠检出 DMR 信号
- `nf + 15dB` 的门限经验值：DMR 4FSK 在 12.5kHz 带宽内的功率通常比噪声基底高 18~25 dB
- `distance=20` 对应 ~12.2 kHz，防止同一宽带信号被重复检测

**ETSI 协议关联**：DMR 信道间隔为 12.5 kHz（§5.1），每个物理信道占用 12.5 kHz 带宽。

---

#### Stage 2: 数字下变频 + 抽取 + 信道滤波

**目标**：将目标信号从 2.5 MHz 降到 48 kHz 基带（SPS=10），并滤除邻道干扰。

**输入**：
- 宽带 IQ + 目标频率偏移 `f_offset`

**处理流程**：

```
宽带 IQ (Fs=2.5MHz)
    │
    ▼
复数混频: iq × exp(-j×2π×f_offset×t)
    │  将目标信号搬到 DC
    ▼
有理数重采样: resample_poly(UP=12, DOWN=625)
    │  2.5MHz × 12/625 = 48kHz  ← 精确整数比
    │  内部: upsampling → FIR 抗混叠 → decimation
    ▼
线性相位 FIR: firwin(101, cutoff=6500Hz, fs=48000) → filtfilt
    │  filtfilt = 零相位滤波（前向+反向），无群延迟失真
    ▼
基带 IQ (Fs=48kHz, SPS=10)
```

**输出**：
- `iq_dec` — 48kHz 采样率的基带复信号
- `iq_filtered` — 经 6.5kHz 信道滤波后的信号

**参数计算**：

```
目标 SPS = 4800 symb/s × 10 = 48000 sample/s
重采样比 = 48000 / 2500000 = 12 / 625
```

**为什么这样设计**：
- `resample_poly` 利用有理数比值精确重采样，避免分数 SPS 带来的定时漂移
- `filtfilt` 零相位滤波：FIR 的群延迟在通带内是线性的，正向滤波引入延迟 τ，反向滤波再引入 τ，净延迟为零。这对后续的样点域定帧至关重要——同步码的绝对样点位置不会被滤波器偏移
- 6.5kHz 截止频率：覆盖 4FSK 的 ±1944 Hz 主瓣，同时抑制邻道（12.5kHz 间距）

**ETSI 协议关联**：DMR 符号率 4800 symb/s（§5.2.1），SPS=10 意味着每个符号 10 个样点，提供足够的过采样余量用于定时恢复。

---

#### Stage 3a: FM 鉴频

**目标**：从复基带信号中提取瞬时频率（4FSK 解调）。

**输入**：
- `iq_filtered` — 48kHz 信道滤波后的复信号

**处理流程**：

```
iq_filtered (复信号)
    │
    ▼
差分鉴频: yd[n] = angle(iq[n] × conj(iq[n-1]))
    │  angle(·) = arctan2(Im, Re) → 相位差 → 瞬时频率
    ▼
gated DC 消除:
    │  power = |iq|²
    │  active = power > median(power) + 0.15×(mean−median)
    │  cfo_dc = mean(yd[active])
    │  y_centered = yd − cfo_dc
    ▼
粗缩放: y_scaled = y_centered × 3 / (2π × DEV_NOMINAL / Fs)
    │  DEV_NOMINAL = 1944 Hz (外层 ±3 符号频偏)
    │  2π × 1944 / 48000 = 0.2545 rad/sample
    │  标称化: ±1944 Hz → ±3 电平
    ▼
瞬时频率序列 (标称符号刻度)
```

**输出**：
- `y_scaled` — 标称刻度 (±3) 的瞬时频率序列

**为什么这样设计**：
- **Gated DC 消除**是关键创新。传统方法对整个序列取均值来估计 CFO，但 TDMA 空闲段的噪声鉴频值会严重偏置均值。Gated 方法只在有信号功率的区间估计 DC，消除了空闲段的污染
- 门限 `median + 0.15×(mean−median)`：对恒包络 FSK，mean≈median，差值小；对 TDMA（50%占空比），mean 被拉低，差值大，0.15 系数确保只选出真正有信号的部分

**ETSI 协议关联**：DMR 4FSK 外层符号频偏为 ±1944 Hz，内层为 ±648 Hz（§5.2.4），符号率 4800 symb/s。频偏值 D = 1944 Hz ≈ 0.405 × 4800，符合协议对调制指数的要求。

---

#### Stage 3b: Integrate-and-Dump 匹配滤波

**目标**：对 4FSK 阶梯状瞬时频率做匹配滤波，最大化符号时刻的信噪比。

**输入**：
- `y_scaled` — 标称刻度的瞬时频率序列

**处理流程**：

```
y_scaled
    │
    ▼
boxcar 卷积: np.convolve(y_scaled, ones(SPS)/SPS, mode='same')
    │  运行窗口平均，长度 = 10 样点 = 1 符号周期
    ▼
clip: np.clip(y_if, -5, +5)
    │  抑制异常尖峰（FM click）
    ▼
y_if — 积分清洗后的信号
```

**输出**：
- `y_clipped` — 匹配滤波后的信号

**为什么这样设计**：
- 4FSK 的瞬时频率在每个符号周期内是**恒定值**（理想情况下是阶梯状），而非线性变化。boxcar（矩形脉冲响应）是恒定值信号的最优匹配滤波器
- 不使用 RRC（根升余弦）：RRC 匹配的是线性调制的脉冲成形，4FSK 是恒包络非线性调制，RRC 不匹配
- clip 到 ±5 是为了防止 FM click 尖峰（可达 ±22000 Hz → ±34 标称值）在后续的 NCC 相关中淹没正常信号

---

#### Stage 4: 多模板 NCC 同步

**目标**：在鉴频输出中检测 DMR 同步码，区分 BS Sourced 和 MS Sourced。

**输入**：
- `y_clipped` — 匹配滤波后的信号

**处理流程**：

```
模板库:
  "BS Sourced"  voice_sync: 755FD7DF75F7 → 24 符号 → 上采样 ×SPS → 240 样点
  "MS Sourced"  voice_sync: 7F7D5DD57DFD → 24 符号 → 上采样 ×SPS → 240 样点

对每个模板:
    │
    ▼
互相关: corr = signal.correlate(y_clipped, template_wave, mode='same')
    │
    ▼
能量归一化: ncc = corr / sqrt( energy(y_clipped) × energy(template) )
    │  其中 energy = convolve(y², ones(wlen))
    │  归一化到 [-1, 1]，消除信号幅度波动的影响
    ▼
正向峰: find_peaks(+ncc, height=0.68, distance=800)
负向峰: find_peaks(-ncc, height=0.68, distance=800)
    │  distance=800 样点 ≈ 16.7ms，小于最短的突发间隔 30ms
    ▼
NCC 峰列表: [(位置, 极性), ...]
```

**输出**：
- `ncc_results` — 每个模板的 NCC 曲线
- `peaks_found` — 正向峰和负向峰的位置列表

**为什么这样设计**：
- **同时检测正向和负向峰**：信号极性可能反转（IQ 符号翻转），正峰对应同相，负峰对应反相
- `height=0.68`：在 20dB 噪声下，语音同步码的 NCC 峰通常 ≥ 0.85，0.68 远低于正常值但高于噪声段（~0.3-0.5），提供足够的余量
- `distance=800` 样点 = 16.7ms：DMR TDMA 帧中同一时隙的突发最小间隔为 30ms（数据/控制突发），因此 16.7ms 确保不会漏检

**ETSI 协议关联**：
- 语音同步码定义于 §B.2.1。BS Sourced = `755FD7DF75F7`，MS Sourced = `7F7D5DD57DFD`
- 同步码 24 符号全部由 ±3 组成（无 ±1），设计上就是为了最大化相关峰的信噪比

---

#### Stage 5: 协议周期验证

**目标**：利用 DMR TDMA 帧结构的周期性，区分 DMR 信号和 P25/噪声。

**输入**：
- NCC 峰位置列表

**处理流程**：

```
对每个模板的正向峰:
    │
    ▼
计算相邻峰间距 (ms): diff = Δsample / 48000 × 1000
    │
    ▼
周期验证:
    │  正向峰: 期望 360ms 周期（语音超帧 = 6 突发 × 60ms）
    │  负向峰: 期望 30ms 或 60ms 周期（数据/控制突发）
    │
    ▼
判定: diff % target_period < 15ms || diff % target_period > (target_period - 15ms)
    至少 60% 的峰间距符合期望周期 → 确认 DMR
```

**输出**：
- `best` — `{"name": "MS Sourced", "type": "VOICE", "peaks": [...]}`

**为什么这样设计**：
- DMR 语音超帧周期固定为 **360ms**（= 6 突发 × 60ms TDMA 帧），是 DMR 独有的结构特征
- P25 Phase 1 使用不同的帧结构（C4FM 连续传输），不会出现 360ms 周期
- 30ms 周期对应数据/控制信道（每个 TDMA 帧都有数据突发）
- 15ms 容差 = 0.25 个符号周期的定时不确定性，对 48kHz 采样率足够宽松

**ETSI 协议关联**：DMR TDMA 帧结构定义于 §5.3。每个 TDMA 帧 60ms（= 2 × 30ms 时隙），语音超帧 = 6 个 TDMA 帧 = 360ms（§6.2）。

---

#### Stage 6: Voice LC Header 解码

**目标**：精确定位 Voice LC Header 突发，逐层解码 Slot Type → BPTC → FLC，提取链路控制信息。

这是整个系统最复杂的阶段，分 5 个子步骤。

---

##### 6a. 专用宽滤波前端

**目标**：为 LC 解码提供一条独立的、更宽的信号路径。

**与 Stage 2-3 的关键差异**：

| 参数 | Stage 2-3（主路径） | Stage 6a（LC 前端） | 原因 |
|------|-------------------|-------------------|------|
| FIR 截止频率 | **6500 Hz** | **9500 Hz** | 6.5kHz 削掉了外层 ±1944 Hz 符号的能量，导致 ±3 和 ±1 电平模糊 |
| CFO 估计 | 功率门控均值 | 鉴频值 active 区中位数 | 中位数对 FM click 尖峰更鲁棒 |

**输入**：`iq_dec` (48kHz 基带)

**处理流程**：

```
iq_dec
    │
    ▼
Welch 谱峰 CFO 粗估计 → 混频去除残余载波
    │
    ▼
firwin(151, cutoff=9500Hz) → filtfilt   ← 放宽滤波
    │
    ▼
差分鉴频 + active 区中位数 DC 消除
    │  active = amp > median(amp) + 0.3×(mean(amp)−median(amp))
    │
    ▼
标称化缩放
    │
    ▼
y_lc — 宽滤波前端输出
```

**输出**：`y_lc` — 宽滤波前端输出（48kHz 瞬时频率序列）

**为什么 6500→9500 是决定性的**：

```
DMR 4FSK 瞬时频率分布:
    -1944 Hz    -648 Hz    0 Hz    +648 Hz    +1944 Hz
       ←───────┬───────┬───┼───┬───────┬───────→
               │       │   │   │       │
               │       │   │   │       └── FIR 6.5kHz 在此处已滚降 ~3-6dB
               │       │   │   │           → +1944 Hz 的符号被衰减
               │       │   │   │           → +3 电平与 +1 电平无法区分
               │       │   │   │
               │       │   │   └── FIR 9.5kHz 在此处几乎无衰减
               │       │   │       → ±3 符号完整保留
```

**ETSI 协议关联**：DMR 发射的占用带宽应 ≤ 8.5 kHz（§5.2.5）。但接收端需要更宽的通带以避免 ISI。9.5 kHz 是 8.5 kHz 的 ~1.12 倍，既保留信号完整性又不引入过多噪声。

---

##### 6b. 样点域 DATA 同步码定位

**目标**：在 48kHz 采样率上以样点精度定位 DATA 同步码（Voice LC Header 使用 DATA 同步）。

**输入**：`y_lc` + DATA 同步码模板

**与 Stage 4 的差异**：

| 维度 | Stage 4 (NCC) | Stage 6b (DATA sync NCC) |
|------|--------------|--------------------------|
| 同步码类型 | 语音同步 `7F7D...` | 数据同步 `D5D7...` |
| 目标 | 粗检测、统计周期 | 精确定位每个突发 |
| 阈值 | 0.68 (宽松) | 0.55 (更宽松，确保不漏检) |

**处理流程**：

```
y_lc
    │
    ▼
NCC 与 DATA 同步模板 (BS/MS Sourced 各一个):
    ncc = correlate(y_lc, template_wave) / sqrt(energy(y_lc) × energy(template))
    │
    ▼
find_peaks(abs(ncc), height=0.55, distance=800)
    │
    ▼
去重: 按 round(sc/50) 分组，相邻 50 样点内只保留一个
    │
    ▼
同步候选列表: [(sync_center_sample, polarity_sign), ...]
```

**输出**：`find_data_sync_positions()` 返回 `[(sync_center, sgn), ...]`

**为什么在样点域而不是符号域**：这是整个项目中最重要的方法论转变。符号域需要先做定时恢复（Gardner 等），但定时恢复在噪声下会崩溃。样点域直接用原始采样点做相关，避免了定时恢复的误差传播。

---

##### 6c. 亚符号相位扫描 + 同步码标定

**目标**：对每个 NCC 定位的候选，扫描亚符号相位，找到使 4 电平星座最清晰的相位，同时用同步码做仿射校准。

**输入**：`y_lc` + `sync_center` + `sgn`

**处理流程**：

```
Burst 布局 (132 符号):
  符号  0..48  │ 49..53 │   54..77    │ 78..82 │  83..131
  位   0..97   │98..107 │ 108..155    │156..165│ 166..263
  ┌───────────┬────────┬─────────────┬────────┬───────────┐
  │  Info ①   │Slot ①  │   SYNC      │Slot ②  │  Info ②   │
  │  49 sym   │ 5 sym  │   24 sym    │ 5 sym  │  49 sym   │
  └───────────┴────────┴─────────────┴────────┴───────────┘
                           ↑
                     sync_center 指向符号 66（同步码中心）

对 33 个候选相位 ph ∈ [-4, +4]:
    │
    ▼
start = sync_center − (54+12)×SPS + ph    ← 回溯到符号 0
pos   = start + arange(132) × SPS         ← 132 个符号的样点位置
    │
    ▼
线性插值: seg = sgn × interp(y_lc, pos)   ← 分数样点重采样
    │
    ▼
取同步区: sy = seg[54:78]                 ← 24 个同步符号
    │
    ▼
最小二乘标定: ref = a × sy + b            ← ref 已知 (±3 序列)
    │
    ▼
校准整个突发: segc = a × seg + b
    │
    ▼
量化到 4 电平: near = levels[argmin(|segc − levels|)]
    │
    ▼
残差: resid = mean((segc − near)²)
    │
    ▼
选 resid 最小的相位 → best = (resid, segc, a)
```

**输出**：
- `segc` — 132 个校准后的符号值
- `resid` — 4 电平拟合残差（质量指标）

**为什么这样设计**：
- **33 步 × 0.25 样点步长 = ±4 样点搜索范围**：NCC 的整数样点精度 ±0.5 样点，亚符号扫描将其提升到 ±0.125 样点（1/4 样点）
- **同步码做标定的优势**：24 个符号全是 ±3（已知真值），是最理想的训练序列。相比盲估计（如 percentile），已知真值的仿射拟合精度高得多
- **残差作为质量门控**：如果 resid > 阈值，说明这个 NCC 峰可能是假峰或信号质量太差，可以直接丢弃，避免无意义的 Golay 校验

**ETSI 协议关联**：DATA 同步码 `D5D7F77FD757`（MS Sourced）定义于 §B.2.2。同步码位于突发的符号 54-77，共 24 符号 48 bit（§B.1.1）。

##### 6d. 自适应四电平判决

**目标**：将 132 个校准后的符号值判决为 264 bit。

**输入**：`segc` — 校准后的 132 符号

**处理流程**：

```
segc (132 个符号值)
    │
    ▼
统计百分位数:
    hi   = percentile(segc, 90)     ← 上包络（接近 +3 簇）
    lo   = percentile(segc, 10)     ← 下包络（接近 −3 簇）
    │
    ▼
计算三个判决门限:
    center = 0.5 × (hi + lo)        ← 正负分界
    umid   = 0.5 × (hi + center)    ← +3 与 +1 分界
    lmid   = 0.5 × (lo + center)    ← −1 与 −3 分界
    │
    ▼
逐符号判决:
    v ≥ umid   → [0, 1]  (+3)
    v ≥ center → [0, 0]  (+1)
    v ≥ lmid   → [1, 0]  (−1)
    else       → [1, 1]  (−3)
    │
    ▼
264 bit (bitarray)
```

**输出**：`ba` — 264 bit 的 bitarray

**为什么这样设计**：
- **百分位数而非固定门限**：即使经过仿射校准，不同突发的增益/偏置仍有微小差异（温度漂移、AGC 响应差异）。P10/P90 能自适应跟踪每个突发的实际电平分布
- **为什么是 P10/P90 而不是 min/max**：min/max 对异常值（FM click 残余）极度敏感，一个尖峰就能把门限拉偏。P10/P90 剔除了首尾各 10% 的离群值，更稳健

**ETSI 协议关联**：4FSK dibit 到符号的映射定义于 §5.2.4：
- `01` → +3 (+1944 Hz)
- `00` → +1 (+648 Hz)
- `10` → −1 (−648 Hz)
- `11` → −3 (−1944 Hz)

##### 6e. 协议逐层校验与 FLC 解析

**目标**：对 264 bit 突发做 Slot Type 校验 → BPTC 解交织 → RS 校验 → FLC 解析。

**输入**：264 bit (`ba`)

**处理流程**：

```
ba (264 bit)
    │
    ▼
字段切片:
    slot_type = ba[98:108] + ba[156:166]     ← 20 bit
    info      = ba[0:98]   + ba[166:264]     ← 196 bit
    │
    ▼
第 1 层 — Golay(20,8,7) 校验:
    │  Golay2087.check(slot_type)
    │  提取 CC = slot_type[0:4], DataType = slot_type[4:8]
    │  要求 DataType == 0001 (Voice LC Header)
    │  失败 → 丢弃
    ▼
第 2 层 — BPTC(196,96) 解交织 + 纠错:
    │  BPTC19696.deinterleave_data_bits(info, repair_if_necessary=True)
    │  输出 96 bit = 12 bytes
    ▼
第 3 层 — Reed-Solomon(12,9,4) 校验:
    │  ReedSolomon1294.check(data12, mask=0x969696)
    │  验证 72 bit FLC + 24 bit RS parity 的一致性
    │  失败 → 丢弃
    ▼
第 4 层 — FLC 解析:
    │  FullLinkControl.from_bits(decoded[0:96])
    │  提取: FLCO, FID, ServiceOptions, Group/Target Address, Source Address
    ▼
LC Header 结果: {
    ok, flco, flco_name, fid, fid_name,
    dst_id, src_id, color_code, data_type,
    golay_ok, rs_ok
}
```

**输出**：LC Header 解析结果字典

**为什么三层校验**：
- Golay 是**最外层**：校验 Slot Type（CC + Data Type），20 bit 中 8 bit 信息 12 bit 校验。如果这层都过不了，说明同步位置就是错的
- BPTC 是**中间层**：块积 Turbo 码，196 bit → 96 bit，有强大的纠错能力（可纠正多 bit 错误）。这是 DMR 抗噪声的核心编码
- RS 是**最内层**：校验 FLC 的完整性。RS(12,9,4) over GF(256) 可纠正 1 个符号（8 bit）错误或检测 3 个符号错误

三层独立校验 = 三层信号质量验证，任何一层失败都能精确定位问题所在。

**ETSI 协议关联**：
- Slot Type + Golay 编码：§B.3.1, §B.3.2
- BPTC(196,96)：§B.3.3
- RS(12,9,4)：§B.3.6，Voice LC Header 专用 mask = `0x969696`（§B.3.12）
- FLC 结构：§7.1.1

## 5. 流水线 C：Late Entry 中途加入

### 5.1 业务场景

**LC Header 解码**（流水线 A）只在呼叫建立时出现。如果接收机在呼叫中途加入（late entry），无法收到 LC Header，必须从语音超帧的嵌入信令（Embedded Signalling, EMB）中重组 LC。

### 5.2 协议原理

ETSI TS 102 361-1 §7.1.2 定义了嵌入信令机制：

```
语音超帧 = 6 个 TDMA 帧 = 360ms

  TDMA Frame A    B    C    D    E    F
  ┌─────┬─────┬─────┬─────┬─────┬─────┐
  │ A   │ B   │ C   │ D   │ E   │ F   │  ← 每帧 60ms
  └─────┴─────┴─────┴─────┴─────┴─────┘
     ↑     ↑     ↑     ↑     ↑     ↑
   语音   嵌入   嵌入   嵌入   嵌入   嵌入
   同步   信令   信令   信令   信令   信令

每个语音突发中心携带 2×EMB 块，每块保护 16 bit 头 + 32 bit 信令:
  ┌──────┬──────────┬──────┐
  │ EMB1 │  Sig 32  │ EMB2 │  ← 突发中心 48 bit
  │ 8bit │   bit    │ 8bit │
  └──────┴──────────┴──────┘

4 个连续语音突发的 Signalling 拼接:
  Sig(B) + Sig(C) + Sig(D) + Sig(E) = 128 bit
      │
      ▼
  VBPTC(128,72) 纠错 → 72 bit LC + 5 bit CS5
```

### 5.3 处理流程

```
y (48kHz 鉴频输出)
    │
    ▼
find_voice_sync_anchor(y, name):
    │  NCC 与语音同步模板 → find_peaks(height=0.7)
    │  返回: [(anchor_center, sgn), ...]
    │  每个 anchor = 一个超帧的 Burst A
    ▼
对每个 anchor:
    │
    ▼
lock_phase_from_anchor(y, anchor_center, sgn, name):
    │  亚符号相位扫描 (49步, ±6样点) → 仿射标定 → 选最佳相位
    │  返回: best_phase
    ▼
decode_one_superframe(y, anchor_center, sgn, name, phase):
    │
    ├─→ j=0 (Burst A): 语音同步，跳过（不含嵌入信令的 LC 碎片）
    │
    ├─→ j=1..6 (Burst B~F): 遍历 6 个语音突发
    │       │
    │       ▼
    │   recover_voice_burst(y, anchor_center, j, phase, sgn):
    │       用 anchor 的相位/极性，固定偏移取第 j 个突发
    │       start = anchor_center + BURST_STRIDE(2880) × j − ...
    │       自适应判决 → 264 bit
    │       │
    │       ▼
    │   parse_emb_center(ba264):
    │       取 bit[108:156] → EMB(16bit) + Signalling(32bit)
    │       │
    │       ▼
    │   EmbeddedSignalling.from_bits(emb_bits):
    │       QR(16,7,6) 校验 + LCSS 状态提取
    │       │
    │       ▼
    │   LCSS 状态机:
    │       ┌──────────────────────────────────────────┐
    │       │  等待 FirstFragmentLC                     │
    │       │      │                                    │
    │       │      ▼                                    │
    │       │  收集 Signalling[0]                       │
    │       │      │                                    │
    │       │      ▼                                    │
    │       │  等待 ContinuationLC ─→ 收集 Sig[1]       │
    │       │      │                                    │
    │       │      ▼                                    │
    │       │  等待 ContinuationLC ─→ 收集 Sig[2]       │
    │       │      │                                    │
    │       │      ▼                                    │
    │       │  等待 StopLC ─→ 收集 Sig[3]               │
    │       │      │                                    │
    │       │      ▼                                    │
    │       │  拼接 128 bit ─→ VBPTC(128,72)            │
    │       └──────────────────────────────────────────┘
    │
    ▼
VBPTC(128,72) 解码:
    │  VBPTC12873.deinterleave_data_bits(b128, include_cs5=True)
    │  输出: lc77 = 72 bit LC + 5 bit CS5
    │
    ▼
CS5 校验 (FiveBitChecksum):
    │  verify(lc72, rx_cs5)
    │  rx_cs5 > 30 → 直接拒绝（非法值，bit 翻转）
    │  失败 → 丢弃整个超帧
    ▼
FullLinkControl.from_bits(lc77):
    │  解析: FLCO, FID, SRC, DST
    ▼
Late Entry LC 结果: {
    anchor, cs5_ok, flco, flco_name,
    fid, fid_name, src_id, dst_id
}
```

### 5.4 关键设计决策

#### 5.4.1 为什么用语音同步（Burst A）做锚点而不是数据同步

- 语音同步在超帧中**固定出现**（每 360ms 一次），是可预测的时间基准
- 语音同步码 NCC 峰值更高（0.85+ vs 0.55+），定位更可靠
- Burst A 不含嵌入信令的 LC 碎片，但提供了整个超帧的**相位锁定基准**

#### 5.4.2 为什么语音突发只用自适应判决而不用仿射标定

- Burst A（语音同步）有已知 ±3 符号可用于标定
- Burst B-F 的嵌入信令区域没有已知训练序列——中心 48 bit 全部是 EMB + Signalling，不可预知
- 因此对 B-F 突发只能用自适应百分位数判决，依赖的是**同一超帧内信道条件缓变**的假设

#### 5.4.3 EMB_TOLERANT 模式

```python
EMB_TOLERANT = True   # QR(16,7,6) 失败也收集
```

**设计理由**：QR(16,7,6) 只保护 16 bit 的 EMB 头部，纠错能力很弱（d=6，最多纠正 2 bit 错误）。在 20dB 噪声下 QR 经常失败，但 32 bit Signalling 可能仍然是好的。容错模式下依赖后续的 **CS5 校验**（保护全部 72 bit）做最终过滤，比 QR 强得多。

#### 5.4.4 CS5_STRICT 模式

```python
CS5_STRICT = True    # 要求 CS5 通过
```

**设计理由**：CS5（FiveBitChecksum, §B.3.11）保护全部 72 bit LC。4 个碎片各 32 bit 拼接时，一个 bit 错位就会导致整个 LC 面目全非。CS5 是唯一保护完整 LC 的校验码——QR 只保护 16 bit 头，VBPTC 的纠错也不保证 100% 正确。CS5 是最后一道防线。

## 7. 设计决策记录

### 决策 1：为什么用 `filtfilt` 而不是 `lfilter`

| 方案 | 群延迟 | 对定帧的影响 |
|------|--------|-------------|
| `lfilter` (单向) | FIR 线性相位 → 固定延迟 N/2 样点 | 同步码位置偏移，需补偿 |
| `filtfilt` (双向) | **零**净延迟 | 同步码位置不变 |

选择 `filtfilt` 的原因：样点域 NCC 定帧要求同步码的绝对样点位置准确。任何滤波器引入的延迟偏移都会直接转化为边界定位误差。

### 决策 2：为什么 I&D 匹配滤波而不是 RRC

4FSK 是**恒包络非线性调制**，瞬时频率是阶梯函数，不是线性调制的脉冲串。RRC 匹配的是线性调制（如 QPSK）的升余弦脉冲，对 4FSK 不匹配。Boxcar 是阶梯函数的最优匹配滤波器。

### 决策 3：为什么 LC 前端用独立的宽滤波路径

Stage 2-3 的 6.5kHz 滤波器是为 NCC 同步优化的——同步码全是 ±3 符号，窄滤波不会影响相关峰。但符号解调需要区分 ±3 和 ±1，6.5kHz 在 ±1944 Hz 处已有显著衰减，导致 ±3 和 ±1 电平不可区分。LC 前端用 9.5kHz 独立路径解决这个矛盾。

### 决策 4：为什么在样点域而不是符号域做帧同步

| 域 | 前提条件 | 噪声下的稳定性 |
|----|---------|--------------|
| 符号域 | 需要先做定时恢复（Gardner 等） | 定时恢复在噪声/空闲段崩溃 → 误差传播 |
| 样点域 | 直接对原始采样做 NCC | NCC 有 23.8 dB 处理增益 → 鲁棒 |

样点域方法将定时恢复**推迟**到帧同步之后——先找到信号在哪，再在信号段内做精细的定时恢复。这避免了"在噪声中找符号"的鸡生蛋问题。

### 决策 5：为什么用百分位数 (P10/P90) 而不是聚类做自适应判决

- **聚类**（如 K-means=4）：计算成本高，对初始化敏感，在 20dB 噪声下可能收敛到错误中心
- **百分位数**：O(n log n) 排序即可，P10/P90 对离群值鲁棒，132 个符号的统计量足够稳定

### 决策 6：Late Entry 为什么需要 CS5 而 LC Header 不需要

- **LC Header** 的 FLC 由 RS(12,9,4) 保护，RS 本身就校验了完整 FLC 的完整性
- **Late Entry** 的 LC 来自 4 个碎片拼接。VBPTC(128,72) 只保证码字级纠错，不保证拼接正确。如果某个碎片的 bit 错位（LCSS 状态机误判），VBPTC 解码仍会"成功"但输出垃圾。CS5 校验全部 72 bit，是唯一能检测拼接错误的机制
